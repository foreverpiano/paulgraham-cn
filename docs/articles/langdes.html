语言设计的五个问题

May 2001 · 阅读原文

2001年5月

（这些是我为2001年5月10日在MIT举办的一场编程语言设计专题讨论会准备的笔记。）

1. 编程语言是给人用的。

编程语言是人与计算机交流的方式。只要语言没有歧义，计算机对用什么语言都无所谓。我们之所以需要高级语言，是因为人无法直接应对机器语言。编程语言的核心作用，就是防止我们脆弱的人脑被海量细节所淹没。

建筑师知道，有些设计问题比其他问题更具个人性。最干净、最抽象的设计问题之一是设计桥梁。这时你的工作主要就是用最少的材料跨越给定的距离。而另一端则是设计椅子。椅子设计师必须把时间花在思考人类屁股上。

软件也是如此。为网络设计数据路由算法是一个漂亮的抽象问题，就像设计桥梁一样。而设计编程语言则像设计椅子：完全是在应对人类的弱点。

我们大多数人都讨厌承认这一点。对我们多数人来说，设计具有极高数学优雅性的系统，听起来比迎合人类弱点要有吸引力得多。数学优雅性有它的作用：某些优雅能让程序更容易被理解。但优雅本身不是目的。

当我说语言必须针对人类的弱点来设计时，并不是说要为水平很差的程序员设计语言。事实上，我认为你应该为设计理念设计语言，但即使是最优秀的程序员也存在局限。我不认为有人会喜欢在一个所有变量都是字母x加整数下标的语言里编程。

2. 为自己和朋友设计。

如果回顾编程语言的历史，很多最好的语言都是其作者为自己使用而设计的，而很多最差的语言则是为其他人设计的。

当语言是为其他人设计时，总是针对某一特定群体：那些不如语言设计师聪明的人。于是你就得到了一种居高临下跟你说话的语言。Cobol 是最极端的例子，但很多语言都弥漫着这种精神。

这和语言有多抽象毫无关系。C 相当底层，但它是为自己的作者设计的，这就是黑客喜欢它的原因。

为差劲的程序员设计语言的理由是，差劲的程序员比优秀的程序员多。这可能是事实。但那少数优秀的程序员写出的软件却占了不成比例的巨大份额。

我感兴趣的问题是：如何设计一种连最顶尖的黑客都会喜欢的语言？我碰巧认为这和“如何设计一门好的编程语言”是同一个问题，但即便不是，它至少也是一个有趣的问题。

3. 给程序员尽可能多的控制权。

许多语言（尤其是那些为其他人设计的语言）带着家庭教师的态度：它们试图阻止你做它们认为对你不好的事情。我喜欢相反的方法：给程序员尽可能多的控制权。

当我第一次学习 Lisp 时，我最喜欢的一点是它把我当作平等的伙伴。在此之前我学过的其他语言里，存在“语言”和“我用该语言编写的程序”两个分离的实体。但在 Lisp 中，我编写的函数和宏与构成语言本身的那些完全一样。如果我想，我可以重写这门语言。它有着和开源软件同样的吸引力。

4. 追求简洁。

简洁被低估，甚至被嘲笑。但如果你去看看黑客的内心，就会发现他们其实非常热爱简洁。你听过多少次黑客深情地谈起，比如在 APL 里，只用几行代码就能做出惊人的事情？我认为，真正聪明的人真正热爱的东西，都值得我们关注。

我认为几乎任何能让程序变短的做法都是好的。应该有大量库函数；任何能隐式处理的就应该隐式处理；语法应该简洁到极致；甚至事物的名称也应该简短。

不仅程序应该简短，手册也应该很薄。手册很大一部分篇幅都被澄清、保留意见、警告和特殊情况占用了。如果你强迫自己缩短手册，最好的情况是你通过修复语言中那些需要大量解释的问题来实现这一点。

5. 承认黑客工作的本质。

很多人希望黑客工作是数学，或者至少像自然科学。我认为黑客工作更像建筑。建筑与物理相关，因为建筑师必须设计不会倒塌的建筑，但建筑师的真正目标是建造伟大的建筑，而不是在静力学上做出发现。

黑客喜欢做的事情是编写伟大的程序。而且我认为，至少在我们自己心里，必须记住：写出伟大的程序是一件值得钦佩的事，即使这项工作不容易转化为研究论文这种传统的学术货币。从智力上来说，设计一门程序员会热爱的语言，和设计一门能让你发表论文的糟糕语言，同样有价值。

1. 如何组织大型程序库？

程序库正成为编程语言中越来越重要的组成部分。它们也变得越来越大，这可能很危险。如果找到能完成你想要功能的库函数所需的时间，比自己写它还长，那么所有这些代码就只是在让你的手册变得厚重。（Symbolics 的手册就是个典型例子。）因此我认为我们需要研究如何组织程序库。理想的情况是设计得让程序员能够猜到哪个库调用能做正确的事情。

2. 人们真的害怕前缀语法吗？

这是一个开放问题，我已经思考它很多年了，仍然不知道答案。前缀语法对我来说似乎完全自然，除了可能在数学方面。但 Lisp 不受欢迎的很大原因，可能仅仅是因为它有着不熟悉的语法。如果这是真的，是否要对此采取措施，则是另一个问题。

3. 开发基于服务器的软件需要什么？

我认为，未来二十年里编写的最令人兴奋的新应用中，很多都将是基于 Web 的应用，也就是运行在服务器上、通过 Web 浏览器与你交互的程序。而要编写这类程序，我们可能需要一些新东西。

我们需要的一件事是对基于服务器应用新发布方式的支持。不像桌面软件那样一年一两个大版本，基于服务器的应用是以一系列小改动的形式发布的。你一天可能会有五到十个发布版本。而且通常大家总是会使用最新版本。

你知道如何把程序设计成可调试的吧？同样，基于服务器的软件也必须被设计成可修改的。你必须能够轻松地修改它，或者至少能知道什么是一个小改动，什么是一个重大改动。

另一件对基于服务器的软件可能有用的东西，令人意外的是续延（continuations）。在基于 Web 的软件中，你可以使用类似续延传递风格（continuation-passing style）的东西，在本质上无状态的 Web 会话世界中获得用于 Web 应用的 Lisp的效果。如果成本不是太高，拥有真正的续延也许是值得的。

4. 还有什么新的抽象有待发现？

我不确定这个希望有多合理，但我个人真正想做的一件事，就是发现一种新的抽象——某种能产生像一流函数、递归甚至关键字参数那样大影响的东西。这可能是个不可能实现的梦想。这些东西并不常被发现。但我一直在寻找。

1. 你可以使用任何你想要的语言。

编写应用程序曾经意味着编写桌面软件。而在桌面软件领域，存在一个巨大的偏见：要用和操作系统相同的语言来编写应用程序。所以十年前，写软件几乎就意味着用 C 写软件。后来形成了一种传统：应用程序绝不能用不常见的语言编写。这种传统发展了太长时间，以至于连非技术人员，比如经理和风险投资人也学会了它。

基于服务器的软件彻底打破了这个模型。使用基于服务器的软件，你可以采用任何你想要的语言。几乎还没有人理解这一点（尤其是经理和风险投资人）。少数黑客理解这一点，这就是为什么我们还能听到 Perl 和 Python 这样新的独立语言。我们听到 Perl 和 Python，并不是因为人们用它们来写 Windows 应用。

这对我们这些对设计编程语言感兴趣的人来说，意味着我们的工作现在可能拥有真正的受众了。

2. 速度来自于性能分析器。

语言设计师，或者至少是语言实现者，喜欢编写能生成快速代码的编译器。但我认为这并不是让语言对用户来说快速的原因。Knuth 很早以前就指出，速度只在少数关键瓶颈处重要。而任何尝试过的人都知道，你无法猜出这些瓶颈在哪里。答案是性能分析器（profilers）。

语言设计师在解决错误的问题。用户不需要跑得快的基准测试。他们需要的是能告诉他们自己程序中哪些部分需要重写的语言。这才是实践中速度的真正来源。所以，如果语言实现者把原本花在编译器优化上的一半时间用来编写一个好的性能分析器，也许反而是净得益。

3. 你需要一个应用程序来驱动语言的设计。

这可能不是一条绝对规则，但看起来最好的语言都是和它们被用来编写的某个应用程序一起演化而来的。C 是由那些需要它来进行系统编程的人编写的。Lisp 部分是为了做符号微分而开发的，McCarthy 如此急于开始，以至于他在 1960 年关于 Lisp 的第一篇论文里就已经在写微分程序了。

如果你的应用程序解决的是某个新问题，那就特别好。这会驱使你的语言拥有程序员需要的新特性。我个人有兴趣设计一门适合编写基于服务器应用程序的语言。

[在讨论会上，Guy Steele 也提出了这一点，并额外建议说，这个应用程序不应该包括编写你自己语言的编译器，除非你的语言本来就打算用于编写编译器。]

4. 一门语言必须适合编写一次性程序。

你知道一次性程序是什么：为了某个有限任务而快速编写的程序。我认为如果你环顾四周，会发现很多大型、严肃的程序最初都是作为一次性程序开始的。如果说大多数程序都是从一次性程序起步的，我也不会感到惊讶。因此，如果你想设计一门适合编写通用软件的语言，它就必须适合编写一次性程序，因为那正是大多数软件的幼虫阶段。

5. 语法与语义是相连的。

传统观点认为语法和语义是完全分离的。这听起来会让人震惊，但它们可能并非如此。我认为你想要的语言特性，可能与你表达它的方式有关。

我最近和 Robert Morris 聊天，他指出运算符重载在中缀语法的语言中收益更大。在前缀语法的语言中，你定义的任何函数实际上都是运算符。如果你想为你发明的新型数字定义加法，只要定义一个新函数即可。但如果你在中缀语法的语言中这么做，重载运算符的使用和函数调用在外观上会有很大差异。

1. 新编程语言。

早在 20 世纪 70 年代，设计新的编程语言是很时髦的事。最近则不然。但我认为基于服务器的软件会让新语言重新流行起来。有了基于服务器的软件，你可以使用任何想要的语言，所以如果有人真的设计出一门看起来比现有语言更好的语言，就会有人愿意冒险去使用它。

2. 分时。

Richard Kelsey 在上次讨论中提出这是一个重新到来的想法，我完全同意他的看法。我的猜测（似乎也是微软的猜测）是，大量计算将从桌面转移到远程服务器上。换句话说，分时又回来了。我认为需要在语言层面为其提供支持。例如，我知道 Richard 和 Jonathan Rees 在 Scheme 48 中实现了进程调度，做了大量工作。

3. 效率。

最近似乎开始觉得计算机终于够快了。我们越来越多地听到关于字节码的讨论，这至少对我来说暗示我们觉得 CPU 周期有富余了。但有了基于服务器的软件，我认为情况不会是这样。有人要为运行软件的服务器买单，而每台机器能支持的用户数将是他们资本成本的除数。

因此我认为效率仍然重要，至少在计算瓶颈处。快速进行 I/O 将尤其重要，因为基于服务器的应用要做大量 I/O。

最终字节码可能并不是一个好主意。Sun 和 Microsoft 目前似乎正在进行一场字节码之战。但他们这么做是因为字节码是一个方便插入自身到整个过程中的位置，而不是因为字节码本身是个好主意。整个战场可能最终会被绕过。那会是挺有意思的事。

1. 客户端。

这只是个猜测，但我猜大多数应用的胜出模式将是纯基于服务器的。设计软件时假设每个人都会安装你的客户端，就像设计社会时假设每个人都会诚实一样。这当然很方便，但你必须假设它永远不会发生。

我认为会有各种各样的设备出现，它们具备某种 Web 访问能力，而你唯一能假设的就是它们能支持简单的 HTML 和表单。你的手机上会有浏览器吗？你的 Palm Pilot 里会有电话吗？你的 Blackberry 会换一块更大的屏幕吗？你能在 Gameboy 上浏览网页吗？你的手表呢？我不知道。但如果我押注一切都在服务器上，我就根本不需要知道。把所有另一条前路要稳健得多。

2. 面向对象编程。

我知道这是一个有争议的话题，但我认为面向对象编程没那么了不起。我觉得它对某些需要那种特定数据结构的应用程序来说是个很好的模型，比如窗口系统、仿真和 CAD 程序。但我不明白为什么它要成为所有编程的模型。

我认为大公司里的人喜欢面向对象编程的部分原因是，它能产生大量看起来像工作成果的东西。本来可能自然地表示为一个整数列表的东西，现在可以表示成一个带有各种脚手架和繁文缛节的类。

面向对象编程的另一个吸引力是，方法能给你一部分一流函数的效果。但这对 Lisp 程序员来说已经是旧闻了。当你拥有真正的一流函数时，你可以根据手头任务的需要以任何合适的方式使用它们，而不是把一切都塞进类和方法的模具里。

我认为这对语言设计意味着，你不应该把面向对象编程内置得太深。也许答案是提供更通用、更基础的机制，让人们把他们想要的任何对象系统作为库来设计。

3. 委员会设计。

让你的语言由委员会来设计是一个巨大的陷阱，而且不仅仅是因为大家都知道的那些原因。大家都知道委员会倾向于产生笨重、不一致的设计。但我认为更大的危险是他们不会冒险。当一个人负责时，他可以冒一些委员会永远无法达成一致的风险。

不过，设计一门好语言是否必须冒险呢？很多人可能觉得语言设计应该是相当贴近传统智慧的事情。我打赌这不对。在人们做的其他所有事情中，回报都与风险成正比。为什么语言设计就该不一样呢？
