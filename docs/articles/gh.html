伟大的黑客

July 2004 · 阅读原文

2004 年 7 月

（本文基于 2004 年在 Oscon 大会上的演讲整理而成。）

几个月前我刚完成一本新书，在书评里反复看到“挑衅”“争议性”这样的词，甚至还有“愚蠢”。

我本没打算让这本书引发争议。我只是想让它更高效。我不想浪费读者的时间去讲他们已经知道的东西，直接给出差异（diffs）会更有效率。但我想这必然会产生一本让人不安的书。

爱迪生们

书中哪个观点最具争议性，这一点毫无争议：那就是财富差异可能不像我们想象的那么严重。

我在书中并没有说财富差异本身是好事。我说的是，在某些情况下，它可能是好事的征兆。剧烈的头痛不是好事，但它可能是好事的信号——比如，你在头部受到重击后正在恢复意识。

财富差异可以是生产力差异的信号。（在一个只有一人的社会里，二者完全相同。）而生产力存在差异几乎肯定是好事：如果你的社会完全没有生产力差异，那多半不是因为人人都像 Thomas Edison，而是因为根本没有 Thomas Edison。

在低技术社会中，你看不到太大的生产力差异。如果是一个游牧部落在捡柴生火，最好的捡柴人和最差的相比，生产力能高出多少？最多两倍？而当你给人们一台复杂的工具，比如电脑，他们能用它做的事情差异就巨大无比。

这并不是什么新观点。Fred Brooks 在 1974 年就写过，他引用的那项研究发表在 1968 年。但我认为他低估了程序员之间的差异。他是用代码行数来衡量生产力的：最好的程序员解决同一个问题只需要十分之一的时间。可如果问题本身都没有给定呢？在编程领域，和许多其他领域一样，困难的部分不是解决问题，而是决定要解决什么问题。想象力很难衡量，但实际上它主导了那种用代码行数来衡量的生产力。

各行各业的生产力都存在差异，但很少有哪个行业差异如此之大。程序员之间的差异大到几乎成了“种类”的不同。我不认为这是编程本身固有的特性。在每个领域，技术都会放大生产力的差异。我认为编程领域正在发生的事，只是我们拥有了大量的技术杠杆而已。但每个领域的杠杆都在不断变长，因此我们看到的这种差异，会被越来越多行业在未来看到。企业和国家的成功，将越来越取决于它们如何应对这种差异。

如果生产力的差异随技术进步而增加，那么最有生产力的人不仅贡献会不成比例地大，而且还会随时间不断增长。当一个团队 90% 的产出来自 1% 的成员时，如果有什么东西（不管是维京人的劫掠还是中央计划）把这些人的生产力拉低到平均水平，你就会遭受巨大损失。

如果我们想从这些人身上获得最大价值，就需要理解这些特别有生产力的人。他们受什么驱动？他们需要什么来完成工作？你如何识别他们？你如何吸引他们来为你工作？当然，还有一个问题：你如何成为其中一员？

不仅仅是金钱

我认识少数几位超级黑客，于是坐下来思考他们有哪些共同点。他们最显著的特征，大概就是他们真的热爱编程。普通程序员写代码是为了付账单。伟大的黑客则把它当作自己为了好玩而做的事，而且很高兴发现居然有人愿意为此付钱。

人们有时说伟大的程序员对金钱无动于衷。这并不完全正确。他们真正在乎的确实只是做有趣的工作。但如果你赚到了足够多的钱，你就可以做任何自己想做的事，因此黑客们会被赚到巨额财富的想法所吸引。只要他们还必须每天准时上班，他们就更关心自己在做什么，而不是为此能拿到多少钱。

从经济学角度看，这是一个极其重要的事实，因为它意味着你不必按真正的价值支付伟大的黑客报酬。一个伟大的程序员的生产力可能是普通程序员的十倍甚至百倍，但他会觉得自己能拿到三倍的薪水就已经很幸运了。我后面会解释，这部分是因为伟大的黑客不知道自己有多厉害。但也因为金钱并不是他们最想要的东西。

黑客想要什么？和所有匠人一样，黑客喜欢好的工具。其实这么说还太轻了。优秀黑客会觉得用糟糕的工具简直无法忍受。他们会直接拒绝参与那些基础设施不对的项目。

我曾经工作过的一家初创公司，公告板上钉着一张 IBM 的广告。那是一张 AS400 的图片，我记得标题是“黑客鄙视它。”[1]

当你决定为一个项目采用什么基础设施时，你不只是在做技术决策，也是在做社会决策，而后者可能更重要。比如，如果你的公司想开发一些软件，用 Java 写似乎是个谨慎的选择。但当你选择一门语言时，你同时也在选择一个社区。能雇来做 Java 项目的人，不会像能雇来做 Python 项目的人那么Python 悖论。而你的黑客的质量可能比你选择的语言更重要。虽然老实说，好黑客更喜欢 Python 而不是 Java，这件事本身就足以说明这两种语言的相对优劣了。

商业人士喜欢最流行的语言，因为他们把语言看作标准。他们不想把公司押在 Betamax 上。然而语言的问题在于，它们不仅仅是标准。如果你需要通过网络传输数据，当然应该用 TCP/IP。但编程语言不只是一个格式。它是一种表达媒介。

我读到过 Java 刚刚超过 Cobol 成为最流行的语言。作为一个标准，这已经是最好的了。但作为表达媒介，你可以做得好得多。在我能想到的所有伟大程序员里，我只知道一位会主动选择用 Java 编程。在所有我不为 Sun 工作、也不用 Java 的伟大程序员中，我一个都没见过。

伟大的黑客通常还坚持使用开源软件。不仅因为它更好，还因为它能给他们更多控制权。优秀黑客坚持要有控制权。这正是他们优秀的原因之一：当东西坏了，他们必须把它修好。你希望他们对你让他们写的软件也抱有这种心态。当他们对操作系统也抱有同样心态时，你不应该感到惊讶。

几年前，一位风投朋友告诉我他正在参与的一家新创公司，听起来很有前景。但下次再聊时，他说他们决定把软件构建在 Windows NT 上，并且刚刚雇了一位非常有经验的 NT 开发者担任首席技术官。听到这话时，我心想，这些人完蛋了。首先，这位 CTO 不可能是一流黑客，因为要成为杰出的 NT 开发者，他必须多次主动选择使用 NT，而我无法想象一个伟大黑客会这么做；其次，即使他很优秀，如果项目必须构建在 NT 上，他也很难雇到其他优秀的人。[2]

最后的边疆

在软件之后，对黑客来说最重要的工具大概就是他的办公室。大公司认为办公室空间的功能是体现层级。但黑客使用办公室不仅仅为此：他们把办公室当作思考的地方。而如果你是一家科技公司，他们的想法就是你的产品。因此，让黑客在嘈杂、分散注意力的环境中工作，就像让一个油漆厂在充满煤烟的空气中生产。

漫画《呆伯特》对格子间有许多吐槽，而且理由充分。我认识的所有黑客都讨厌格子间。仅仅是被打断的可能性就足以阻止黑客去解决困难的问题。如果你想在有格子间的办公室里真正完成工作，你只有两个选择：在家工作，或者早来晚走或者周末来，那时候没人。难道公司没意识到这是某事出了问题的信号吗？办公室环境应该是帮助你工作的，而不是你必须克服它才能工作的地方。

像思科这样的公司很自豪地宣称每个人都有格子间，连 CEO 也不例外。但他们并不像自己以为的那么先进；显然他们仍然把办公室空间视为等级的象征。还要注意，思科以几乎不做内部产品开发而闻名。他们通过收购创造了新技术的初创公司来获得新技术——在那里，黑客们大概有安静的地方工作。

有一家大公司真正理解黑客需要什么，那就是微软。我曾经见过微软的一则招聘广告，上面是一张大大的门图片。前提是：来我们这里工作，我们会给你一个真正能完成工作的场所。而且你知道，微软在能够内部开发软件的大公司中非常突出。或许不算特别好，但已经足够好了。

如果公司希望黑客高效工作，就应该看看他们在家里是怎么做的。在家里，黑客可以自己安排一切，以便完成最多的事。而且在家工作时，黑客不会在嘈杂的开放空间工作；他们会在有门的房间里工作。他们在温馨、像社区一样的地方工作，周围有人，需要思考的时候可以出去走走，而不是在广阔停车场里的玻璃盒子里。他们有一张沙发，累了可以躺下午睡，而不是坐在桌前昏昏沉沉地假装工作。不会有拿着吸尘器的人每天晚上在黄金黑客时间段轰隆隆地清扫。不会有会议，更不用说该死的公司团建或团队建设活动了。当你看看他们在电脑上做的事情时，你会发现这印证了我之前关于工具的说法。他们工作时可能必须用 Java 和 Windows，但在家里可以自己选择时，你更可能会发现他们用 Perl 和 Linux。

事实上，那些关于 Cobol 或 Java 是最流行语言的统计数据可能会产生误导。如果我们想知道哪些工具最好，就应该看黑客在可以自由选择时会选什么——也就是说，在他们自己的项目中。当你提出这个问题时，你会发现开源操作系统已经拥有压倒性的市场份额，而第一语言很可能是 Perl。

有趣

除了好的工具，黑客还想要有趣的项目。什么让一个项目变得有趣？显然，像隐形飞机或特效软件这样 overtly sexy 的应用会很有趣。但任何应用只要提出新颖的技术挑战，都可以变得有趣。因此很难预测黑客会喜欢哪些问题，因为有些问题只有在参与者发现一种新的解决方案时才会变得有趣。在 ITA（他们写了 Orbitz 内部的软件）之前，做航空票价搜索的人可能认为这是最无聊的应用之一。但 ITA 通过Carl de Marcken：Orbitz 背后的故事问题，用一种更具野心的方式让它变得有趣。

我认为 Google 也发生了类似的事。Google 成立时，所谓的门户网站之间的共识是搜索既无聊又不重要。但 Google 的那帮人不这么认为，这就是他们能把搜索做得这么好的原因。

这是一个管理者可以发挥作用的领域。就像父母对孩子说，我打赌你十分钟内清理不完整个房间，一个好的管理者有时能把问题重新定义成一个更有趣的问题。Steve Jobs 似乎特别擅长这一点，部分原因仅仅是他有很高的标准。在 Mac 出现之前，已经有很多便宜的小型电脑。他把问题重新定义为：做一台漂亮的。然后这可能比任何胡萝卜加大棒都更能驱动开发者。

他们确实做到了。当 Mac 刚出现时，你甚至不用打开它就知道它很棒；从外壳就能看出来。几周前我在剑桥街上走，在别人的垃圾里看到一个似乎是 Mac 便携包的东西。我打开一看，里面是一台 Mac SE。我把它带回家插上电源，它启动了。快乐的 Macintosh 笑脸，然后是 Finder。我的天，太简洁了。它就像……Google。

黑客喜欢为有高标准的人工作。但仅仅严格还不够。你必须坚持正确的事情。这通常意味着你自己也必须是个黑客。我偶尔读到过关于如何管理程序员的文章。实际上应该有两篇文章：一篇是关于如果你自己就是程序员该怎么办，另一篇是关于如果你不是该怎么办。而第二篇大概可以浓缩成两个字：放弃。

问题主要不是日常管理。真正优秀的黑客几乎是自我管理的。问题是，如果你不是黑客，你就无法分辨谁是优秀黑客。一个类似的问题解释了为什么美国汽车那么丑。我把它称为设计悖论。你可能以为只要雇一个伟大的设计师，就能让你的产品变漂亮。但如果你自己没有好的匠人的品味，你怎么能认出好的设计师呢？从定义上讲，你无法从他的作品集判断。而且你也不能看他获得的奖项或做过的工作，因为在设计领域，和大多数领域一样，这些东西主要由时尚和社交决定，实际能力排在很后面。没有办法绕过去：如果你不知道什么是美，你就无法管理一个旨在生产美的事物的过程。美国汽车很丑，因为美国汽车公司是由品味糟糕的人经营的。

这个国家的许多人认为品味是一种捉摸不定的、甚至是轻浮的东西。它既不是。想要驱动设计，管理者必须是公司产品最挑剔的用户。如果你真的有很好的品味，你就能像 Steve Jobs 那样，把“让你满意”变成优秀人才喜欢解决的那类问题。

讨厌的小问题

说哪些问题不有趣其实很容易：那些不是解决几个大而明确的问题，而是要解决大量讨厌小问题的情况。最糟糕的项目类型之一，就是给一个充满 bug 的软件写接口。另一种是不得不为某个特定客户的复杂且定义不清的需求做定制。对黑客来说，这些项目就像凌迟处死。

讨厌小问题的显著特征是，你无法从中学到任何东西。写一个编译器很有趣，因为它教会你编译器是什么。但给一个充满 bug 的软件写接口什么也教不了你，因为那些 bug 是随机的。[3] 因此，让优秀黑客回避讨厌小问题的不仅仅是挑剔。更是一个自我保护的问题。处理讨厌的小问题会让你变蠢。优秀黑客回避它，就像模特回避芝士汉堡一样。

当然，有些问题天生就具有这种特性。而且由于供需关系，它们报酬特别高。因此，如果一家公司能找到办法让伟大黑客去解决枯燥的问题，它会非常成功。你会怎么做呢？

这种情况发生的一个地方是初创公司。在我们的初创公司里，我们让 Robert Morris 担任系统管理员。这就像让滚石乐队在成人礼上演出。你雇不到这种人才。但人们会为他们自己创办的公司做任意数量的苦活。[4]

更大的公司通过划分公司来解决这个问题。他们通过设立一个单独的研发部门，让聪明人不必直接面对客户的讨厌小问题，从而吸引他们来工作。[5] 在这个模型中，研发部门的功能像一座矿井。他们生产新想法；也许公司的其他部门能用得上。

你可能不必走到这个极端。自底向上编程 提出了另一种划分公司的方式：让聪明人担任工具制作者。如果你的公司开发做 x 的软件，就设一个小组来构建编写这类软件的工具，另一个小组用这些工具来写应用。这样你也许能让聪明人写出 99% 的代码，但仍能让他们几乎像在传统研发部门一样与用户隔绝。工具制作者会有用户，但那些用户只是公司自己的开发者。[6]

如果微软采用这种方法，他们的软件就不会有那么多安全漏洞，因为写实际应用的不太聪明的人就不会去做分配内存这样的底层工作。他们不会直接用 C 写 Word，而是把 Word 语言的大乐高块拼在一起。（我相信技术术语叫 Duplo。）

抱团

除了有趣的问题，优秀黑客喜欢的是其他优秀黑客。伟大的黑客倾向于抱团——有时惊人地明显，比如在 Xerox Parc。因此，你吸引优秀黑客的数量和你为他们创造的环境好坏不是线性关系。这种抱团倾向意味着它更接近环境的平方。因此这是赢家通吃。在任何时候，大约只有十到二十个地方是黑客最想去工作的，如果你不是其中之一，你不仅会拥有更少的伟大黑客，你会一个都没有。

拥有伟大黑客本身不足以让一家公司成功。这对目前两个热点 Google 和 ITA 效果很好，但对 Thinking Machines 或 Xerox 没有帮助。Sun 曾经风光过一段时间，但他们的商业模式是向下电梯。在那种情况下，即使最好的黑客也救不了你。

不过，我认为在其他条件相同的情况下，一家能吸引伟大黑客的公司会有巨大优势。有些人会不同意这种观点。当我们在 90 年代拜访风投公司时，好几家告诉我们，软件公司不是靠写出伟大软件取胜，而是靠品牌、渠道控制和正确的交易。

他们似乎真的相信这一点，我想我知道原因。我认为很多风投（至少是下意识地）在寻找的是下一个微软。当然，如果微软是你的榜样，你就不应该寻找那些希望靠写伟大软件取胜的公司。但风投寻找下一个微软是错误的，因为除非有另一家公司准备在恰当时刻弯腰成为下一个 IBM，否则没有任何初创公司能成为下一个微软。

把微软当作榜样是错误的，因为他们的整个文化都来自那一次幸运的突破。微软是一个坏的数据点。如果你把他们排除，你会发现好的产品确实倾向于在市场上获胜。风投应该寻找的是下一个 Apple，或者下一个 Google。

我认为 Bill Gates 知道这一点。他对 Google 的担心不是他们品牌的威力，而是他们拥有更好的黑客。[7]

识别

那么谁是伟大的黑客？你怎么知道自己遇到了一个？结果发现这非常难。即使是黑客也无法分辨。我现在相当确定我的朋友 Trevor Blackwell 是个伟大黑客。你可能在 Slashdot 上读到过他自己做了一台 Segway。这个项目的惊人之处在于他用一天时间（顺便说一句，是用 Python）写完了所有软件。

对 Trevor 来说，这只是常规操作。但我第一次见到他时，我以为他是个彻头彻尾的傻瓜。他站在 Robert Morris 的办公室里，对着他滔滔不绝地说着什么，我记得我站在他身后疯狂地给 Robert 打手势，让他把这个疯子赶出办公室，我们好去吃午饭。Robert 说他一开始也错判了 Trevor。显然 Robert 第一次见到他时，Trevor 刚开始一个新计划，要把生活中方方面面的每件事都写在索引卡上，随身携带。他也刚从加拿大过来，有很重的加拿大口音，还留着 mullet 发型。

这个问题还因为另一个事实而加剧：尽管黑客以社交迟钝闻名，但有时他们会花很多力气让自己显得聪明。我读研究生时偶尔会去 MIT 人工智能实验室晃荡。一开始有点吓人。那里的每个人都说得那么快。但过了一段时间我学会了说得快的技巧。你不需要想得更快；只需要用两倍的词把所有事情说出来。

在信号中有这么多噪音的情况下，遇到黑客时很难判断他们是否优秀。我现在都做不到。你也无法从他们的简历判断。似乎判断一个黑客的唯一方法就是和他一起做一个项目。

这就是为什么高科技区域只出现在大学周围的原因。这里的活性成分与其说是教授，不如说是学生。初创公司在大学周围成长，因为大学把有前途的年轻人聚集在一起，让他们共同做同样的项目。聪明的那些人会认出其他聪明人，然后一起构思他们自己的新项目。

因为除了通过一起工作，你无法判断谁是伟大黑客，所以黑客自己也不知道自己有多厉害。这在大多数领域都在一定程度上成立。我发现那些在某件事上很厉害的人，并不那么确信自己的伟大，而是对为什么其他人显得那么无能感到困惑。

但黑客特别难知道自己有多厉害，因为他们的工作很难比较。这在大多数其他领域更容易。在百米赛跑中，你十秒钟就知道谁最快。即使在数学中，似乎也有一个普遍共识，知道哪些问题是难的，什么算是一个好的解决方案。但黑客就像写作。谁能说两本小说哪一本更好？肯定不是作者自己。

至少对黑客来说，其他黑客能分辨。这是因为，和小说家不同，黑客是在项目上协作的。当你在网上把几个困难的问题扔给一个人，你很快就能知道他们回击得有多猛。但黑客无法观察自己工作。所以如果你问一个伟大黑客他有多厉害，他几乎肯定会回答：我不知道。他不是在谦虚。他是真的不知道。

而我们谁都不知道，除了那些我们真正共事过的人。这让我们处于一个奇怪的境地：我们不知道我们的英雄应该是谁。那些变得著名的黑客，往往是因为公关的随机意外而著名。我偶尔需要举一个伟大黑客的例子，但我从来不知道该用谁。首先想到的名字总是我个人认识的人，但用他们似乎很蹩脚。所以我想，也许我应该说 Richard Stallman，或者 Linus Torvalds，或者 Alan Kay，或者其他类似的名人。但我完全不知道这些人是不是伟大黑客。我从来没有和他们一起做过任何项目。

如果存在一个黑客界的 Michael Jordan，没有人知道，包括他自己。

培养

最后，黑客们一直好奇的问题是：你如何成为一个伟大黑客？我不知道是否可能把自己变成一个。但肯定可以做一些让自己变蠢的事，如果你能让自己变蠢，大概也能让自己变聪明。

成为优秀黑客的关键可能是去做你喜欢的事。当我想到我认识的伟大黑客时，他们的共同点之一是，让他们去做任何不想做的事极其好的拖延与坏的拖延。我不知道这是原因还是结果；可能两者都是。

要把事情做好，你必须如何做你热爱的事它。因此，在你能把黑客活动保持为你热爱的事情的程度上，你就很可能把它做好。试着保持你在 14 岁时对编程的那种惊奇感。如果你担心你现在的工作正在腐蚀你的大脑，那很可能确实如此。

最好的黑客当然往往很聪明，但在很多领域都是如此。有没有什么品质是黑客独有的？我问了一些朋友，他们提到的第一件事是好奇心。我一直以为所有聪明人都好奇——好奇心只是知识的一阶导数。但显然黑客特别好奇，尤其是对事物如何运作。这很合理，因为程序本质上是关于事物如何运作的巨型描述。

几个朋友提到黑客的专注能力——用其中一个的话说，就是“把脑袋以外的一切都调出去”的能力。我当然注意到了这一点。我也听几个黑客说，即使只喝了半杯啤酒，他们就完全没法编程了。所以也许黑客确实需要某种特殊的专注能力。也许伟大黑客能在头脑中加载大量上下文，因此当他们看一行代码时，他们看到的不仅仅是那一行，而是围绕它的整个程序。John McPhee 写到 Bill Bradley 作为篮球运动员的成功部分归功于他非凡的周边视觉。“完美”视力意味着大约 47 度的垂直周边视觉。Bill Bradley 有 70 度；当他看着地板时也能看到篮筐。也许伟大黑客有某种类似的天生能力。（我作弊用了非常简洁就是力量的语言，这缩小了球场。）

这可以解释关于格子间的认知断层。也许负责设施的人，因为没有需要被打断的专注力，完全不知道在格子间工作对黑客来说就像把大脑放进搅拌机。（而 Bill，如果自闭症的传闻是真的，就太清楚了。）

我注意到伟大黑客和一般聪明人之间的一个区别是，黑客更你不能说的话。如果说优秀黑客之间有什么秘密握手，那就是当他们彼此足够了解，可以表达那些会被公众用石头砸死的观点时。我能理解为什么政治不正确在编程中是一种有用的品质。程序非常复杂，而且至少在优秀程序员手中，非常流动。在这种情况下，养成质疑假设的习惯很有帮助。

你能培养这些品质吗？我不知道。但你至少可以不压制它们。所以这里是我最好的尝试配方。如果有可能把自己变成伟大黑客，那么方法可能是和自己达成以下协议：你永远不必做无聊的项目（除非否则你的家人会挨饿），作为回报，你永远不允许自己做半吊子的工作。我认识的所有伟大黑客似乎都做出了这个协议，尽管也许他们对此都没有选择。
