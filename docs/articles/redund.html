Kragen Sitaker：冗余与能力

阅读原文

有人提出的观点。我对这个看法抱有一定认同；如果“能力”一词在谈论编程语言时有任何意义，那么一种更强大的语言，就是能用同样的人力完成更多工作的语言。编程中的人力投入——设计、理解、构建和调试——与源代码的长度密切相关，因此通常来说，更简洁的程序所需投入就越少。简洁就是力量

不过，还有其他因素也会影响所需的人力投入，有些与语言相关，有些则无关。

例如，优秀的错误报告能大幅减少查找和修复问题的时间。这一点的重要性与你犯错的数量以及猜测错误原因的能力成正比——新手会犯很多错误且无法猜测，而超级程序员犯错很少，并且通常很擅长猜测。

错误报告在很大程度上依赖于冗余。Knuth 曾说过[0]，如果 TeX 的语言中拥有更多冗余，编程会变得更容易。Perl 消除了大部分冗余的数据类型转换——程序因此更加简洁，但错误也更容易出现。（在我写的一个 Perl 程序中，我有一个由 LIST 引用组成的列表，表示一个二维数值数组。在遍历这个列表的循环中，我错误地写了 $list[$i] 而不是 $list[$i][$j]，结果程序愉快地用内存地址乘以整数，得到了天文数字般巨大且错误的结果。）像 Ada 和 XML 这样使用不同种类结束标记的语言，通常能准确告诉你漏掉了哪个结束标记，或者多写了一个；而只使用右括号的语言则做不到这一点。

一个随机字符串成为合法程序的可能性越大，就越难做好错误报告。

类型检查也依赖于冗余；如果像 Forth 那样任何运算符都能应用于任何类型，那么你唯一能发现类型错误的方式就是得到错误的答案或程序崩溃；而如果只有基本操作才会检查类型，那么错误报告的位置可能会离实际发生类型错误的地方很远。当我把类型不匹配的值传给 Python 库时，随之而来的错误信息往往需要花一番工夫才能和我自己的代码联系起来。

当然，冗余会降低简洁性。

另一个消耗人力的地方是查阅手册。我用 Python 编程时要花大量时间查资料；例如，我很少能记住 re.sub() 的参数顺序，而 Perl 那种更具特色的语法反而更容易记住。毫无疑问，如果是 Smalltalk 的方法，我会觉得记住参数顺序要容易得多。

pydoc 帮了大忙；能够直接输入 'pydoc re.sub' 而不用在 HTML 手册里翻来翻去，已经是很大的帮助了。如果能在 Emacs 编辑 Python 程序时直接调用 pydoc，那帮助会更大；Emacs 早在 C 语言环境下就长期支持类似功能（M-x man）。

作为微软开发环境粉丝的 Joel Spolsky 最近指出[1]，静态类型能让语法引导的编辑器主动提示方法签名，而微软的开发环境确实做到了这一点。Lisp 开发环境也能实现这个功能（Emacs 的 ilisp-mode 通过 C-c a 或 eldoc-mode 就能做到），但 Python 环境要做到这一点会非常困难。

[0] 对 Donald E. Knuth 的访谈，Advogato，2000-01-25。

[1] Joel on Software，2002-04-25，主题是“SOAP 反弹”。遗憾的是，Joel 太狭隘了，他甚至不知道怎么拼写“dynamic typing”，而是写成了“variants”。
