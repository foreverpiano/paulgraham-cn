编程常见问题

阅读原文

你用什么编辑器？

vi。

我该怎么学习编程？

找一个会编程的朋友，让他帮你搭建一个可以编辑和运行程序的环境。使用他推荐给初学者的语言（很可能是 Python 或 Ruby）。然后买一本 O'Reilly 的书，从头开始跟着学。

在学习编写和运行程序的基本技能的同时，开始思考你真正想编写的具体程序。这会激励你去学习更多知识。

不要一开始就挑太大的问题。比较好的入门方式是拿一个现有的程序，修改它让它实现一些新的功能。

刚开始写的程序会很丑，但不用担心这一点。每个人都是这样。只要坚持写下去，程序会越写越好。

学习过程中，你会发现阅读别人写的程序很有帮助。但只有在自己尝试编程之后，从中得到的收获才会更多。

最后，结交一些喜欢写程序的朋友。他们可以回答你的技术问题；和他们交流能让你产生新的想法；而且他们会是你最初作品的第一批受众。

你为什么建议直接投入编程项目，而不是先仔细规划？

如果你要解决的是一个简单、明确的问题，比如深度优先搜索，那么事先把一切想清楚并没有坏处。但现实中的大多数问题都不是这样。在实际应用中，你一开始通常并不清楚自己真正要解决的问题到底是什么。因此如果你花大量时间提前做规划，最终得到的将是一个非常详尽的、却用来解决错误问题的计划。

对于复杂且定义不清晰的问题，更好的做法是尽快写一个原型，看看它到底有哪些问题，然后相应地调整你对问题的定义。

程序员被要求做规划，往往不是因为问题本身需要，而是因为项目经理需要。也许程序员应该给经理一个明确的选择：你是希望我用让你感觉舒服的方式来解决问题，还是用能得到最佳解决方案的方式？

你为什么老是在说 Lisp？

我经常谈论的话题其实有很多：小公司比大公司做得更好；格子间很糟糕；要成为优秀的黑客必须理解设计；规划被高估了。这些观点看起来没那么显眼，因为很多读者要么没有先入为主的看法，要么已经同意了。

认识到格子间很糟糕不需要付出任何代价。很少有人在办公室风格上存在既得利益。但每个人对自己已经掌握的编程语言都有既得利益，因为（a）学习一门新语言需要大量工作，（b）编程语言会决定你如何思考程序，因此很难想象出比你熟悉的语言更强大的语言。

我知道，贬低别人使用的语言被认为是不礼貌的。但不礼貌并不等于错误。任何语言设计者都必须面对一些尴尬的问题，比如哪些语言更好，以及为什么更好，就像肛肠科医生必须做一些普通人做会显得极其不礼貌的事情一样。

面向对象编程不是天生就适合某些问题吗？

是，也不是。很多看似是面向对象的问题，如果你能随意使用那些共同构成“面向对象性”的各种概念，就会发现它们其实并非如此。

比如，如果我在写一个 CAD 程序或模拟程序，我可能会使用 OO 抽象（尽管我最后可能会用宏创建自己的 OO 模型，而不是使用语言自带的那个）。

但如果我试图解决一位读者发给我的、被当作 OO 问题经典范例的问题，我就不会这么做。

假设你有 n 个串口，每个串口可能使用 k 种协议中的一种，而且必须能在运行时进行配置。

我只会用一个 n 行 k 列的闭包数组来表示它。
