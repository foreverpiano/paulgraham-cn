LFM 和 LFSP

阅读原文

（这是一封 Mike Vanier 发来的邮件。它对程序员面临的困境描述得如此到位，我觉得应该让大家都看看。）

我的博士研究课题是编写非常复杂的神经系统仿真程序。我使用的是一个用 C 语言开发的内部仿真工具包，它自带一套脚本语言，全都是团队内部开发的。我对它进行了大量扩展，但那段经历实在太痛苦了，我觉得我再也不想参与任何大型 C 项目了。

既然我还想继续在这个领域工作，也热爱 hacking，我就想把这个仿真器“以正确的方式”重写一遍。然而，我一直在语言选择上犹豫不决。很明显，核心仿真对象必须用 C++ 编写。C 太痛苦了，而其他语言在速度上都会带来不可接受的损失（仿真这个领域极其罕见地属于“速度永远不嫌多”的类型）。但这部分代码可能不到整个项目的 50%，甚至可能少得多。剩下的都是基础设施：脚本接口以及大量支持代码。

脚本语言我想用 Scheme 或某种 Lisp 方言。但基础设施该用什么语言还不清楚。我可以用 C++，但这毫无必要地痛苦，尤其是基础设施对速度并不敏感。所以我决定用 Java：它足够快，有大量现成库，而且很多人都会用，我将来也比较容易找到其他人一起开发。做出这个决定后，我的兴趣却渐渐消退，于是我开始了另一个无关的项目。

在做那个新项目（它同时用到了 Scheme 和 Objective Caml（Ocaml，一种 ML 方言））的过程中，我突然意识到，用 Ocaml 来实现中间层会比 Java 好得多。它更快、类型检查更严格、语言能力强大得多，而且因为有类型推断和交互式 REPL，甚至可以直接作为自己的脚本语言。如果需要，我也能轻松地在 Ocaml 之上实现一个简单的类 Lisp 语言。Ocaml 的 C 接口也相当成熟，还有一个规模可观的标准库（虽然远不如 Java 那样庞大）。而且它比 Java 轻量得多。但最重要的理由是：用它编程比用 Java 有趣得多。

写 Java 代码虽然不像写 C 那样痛苦（核心转储之类的），却会让我昏昏欲睡。而写 Ocaml（如果真有一门“为聪明人设计的语言”，那一定就是它）则令人兴奋。我对这个项目的热情一夜之间提升了三倍。有趣的问题是：为什么 Ocaml 比 Java 有趣得多？为什么“为聪明人设计的语言”（LFSPs）比“为大众设计的语言”（LFMs）编程起来要愉快得多？

一种可能是，LFSPs 通常比较小众，因此显得更新奇。我承认这也是部分原因，但它没有说到点子上。任何新语言都会让人感到新鲜，但新鲜感通常很快就会消退。真正的关键在于，LFSPs 对抽象的支持要强大得多，尤其是允许程序员定义自己的抽象，这一点远胜于 LFMs。

这并非偶然；LFMs 故意限制了语言的抽象能力，因为它们认为用户“驾驭不了”那么大的威力。这就形成了一层抽象的天花板：你的设计只能抽象到这个程度，不能再进一步。这让普通程序员 Joe Average 感到安心，因为他知道自己不会遇到看不懂的代码。这也让老板 Joe Boss 感到安心，因为他知道即使把你炒了，也能再雇一个程序员来维护和扩展你的代码。但这对天才黑客 Joe Wizard Hacker 来说却无比痛苦，因为他明明知道自己的设计可以变得 N 倍更通用、更抽象，但语言却迫使他一遍又一遍地做同样的老套事情。

久而久之，这会把人彻底磨垮；如果我每次在 C 里写“for (i = 0; i < N; i++)”都能拿到五分钱，我早就成百万富翁了。我认识好几位程序员，仅仅经过几年高强度 hacking 就彻底 burnout，他们说自己再也不想写代码了。这真的很可悲，我认为其中一个重要原因就是他们在使用 LFM，而本该使用 LFSP。
