首要任务：核心语言

February 2008 · 阅读原文

2008年2月

我目前对 Arc 的首要任务是核心语言——那些既非 car、cdr 这样的基本操作，也非特定用途库函数的运算符。我指的是类似 CL 的 mapcar、let、remove-if-not 之类的运算符。

正如我在 接受 Arc 挑战 中所说，从公理出发通往日常编程完整语言的道路，至少存在一条最优路径，而 Arc 的目标就是尝试找到它。

我并不反对修改公理。如果能以有用的方式做到这一点，那就再好不过了。但这类想法不是每天都能想到的。

我之所以把注意力集中在公理和库函数之间的这个区域，是因为从程序员的角度来看，这些运算符就是语言本身。你的程序正是由它们构成的。如果把 Lisp 比作一栋房子，这些运算符就是前门、客厅沙发和厨房餐桌。它们设计上的微小差异，就能极大地影响这门语言的实际表现。

我因为专注于这一点而饱受批评，但我认为这是一个既重要又困难的问题。它难度的证据之一是：尽管 Lisp 已经快50岁了，但我所知道的所有 Lisp 方言中的核心运算符都远未达到最优。以 cond 宏为例，从 Lisp 1 开始它就带着那些多余的括号，却没人去改进它。我不会惊讶于是否已经有人发现了 Arc 把 if 和 cond 合并为一个运算符的技巧，但它显然不是常识。当我意识到这一点可以做到时，我感到很高兴。

Lisp 核心演化如此缓慢的一个原因是，我们已经习惯了它们。你开始用已有的运算符来思考问题。要想象如何用尚不存在的运算符来重写代码，需要有意识的努力。

这也是我为什么特别关注代码尺寸的原因。长度是一种外部约束。如果你开始看着代码思考“它的长度下限是多少？”，你就离发现那个能让它变得如此简短的新运算符只差一步了。

Lisp 核心演化缓慢的另一个原因是，这更多是一个 设计与研究 问题而非实现问题，而大多数黑客既没有能力也没有意愿去从事这类工作。它不会让你的黑客朋友们印象深刻；不会帮你拿到终身教职；你不再是每天写出500行代码，而是如果能写出-20行代码就觉得自己很幸运。但这绝不是一项无聊的工作。事实上，我认为 Lisp 之所以能持久存在这么久，其中一个原因就是 McCarthy 是一位伟大的设计师。在那个时代，其他人要么创造出被当时硬件所限制的语言，要么用考虑不周的方式试图让编程语言像自然语言，而他却做出了一个精小而永恒的东西。Lisp 的许多优点都来自于那份 brilliance，它穿透后来才华较弱的设计者们留下的迷雾，依然闪闪发光。

有些人在 Arc 最初发布时对其冷嘲热讽，开玩笑说它是“McCarthy 的第二次降临”。我相信他们不是这个意思，但事实上，我确实想从他停下的地方重新开始他曾做过的那种语言设计。

所以我想请求 Arc 用户帮个忙：当我以看似随意、细微的方式打破和修改某些东西时，希望你们能保持宽容。我现在认为整个核心都是用铅笔写成的，希望能够对它进行大量修改。我知道这会让在 Arc 之上编写程序变得很痛苦，但最终的结果应该是值得拥有的。

虽然 Arc 包含了一些用于编写 web 应用的便利库——这些库强大到用一个需要在服务器上保存字符串的问题来比较 Arc 和其他语言都显得像作弊——但这目前并非重点。我写所有这些 web 相关的东西，只是为了制造真实应用对核心产生的压力。因此，虽然我对 web 应用的新想法感兴趣，但我真正想听到的，是关于语言核心的新东西。

在这个高度压缩且已被反复探索的领域，想出新想法更加困难，但这项工作也极具回报。我们知道我们所寻找的东西一定存在。根据定义，从公理到完整语言必然存在某条最优路径。Lisp 的柏拉图式理想形式就藏在那块大理石里。我们所要做的，就是不断凿刻，直到把它呈现出来。
