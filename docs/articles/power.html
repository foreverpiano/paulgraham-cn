简洁就是力量

May 2002 · 阅读原文

2002 年 5 月

“代数符号将大量含义压缩到狭小空间，这是我们借助它们进行推理的另一便利之处。”

- Charles Babbage，引自 Iverson 的图灵奖演讲

在 LL1 邮件列表讨论 书呆子的复仇 引发的议题时，Paul Prescod 写下了一段让我印象深刻的话。

Python 的目标是规范性和可读性，而不是简洁。

表面上看，这对一门编程语言来说是个相当严厉的指责。据我所知，简洁就等于力量。如果真是这样，那么替换之后就变成

Python 的目标是规范性和可读性，而不是力量。

而这个交易（如果算得上是交易的话）似乎不是你想要的。这几乎等于在说 Python 的目标不是成为一门高效的编程语言。

简洁是否等于力量？这在我看来是个重要问题，或许是对语言设计感兴趣的人最重要的问题，而且值得我们正面去面对。我目前还不能确定答案是否简单肯定，但它似乎是个很好的出发假设。

假设

我的假设是：简洁就是力量，或者非常接近，以至于除非在病态的例子中，你都可以把两者视为等同。

在我看来，简洁正是编程语言的存在意义。计算机同样乐于直接接收机器语言指令。我认为我们之所以费力开发高级语言，主要是为了获得杠杆效应，这样我们就能用高级语言的 10 行代码表达（更重要的是思考）原本需要 1000 行机器语言才能表达的内容。换句话说，高级语言的核心作用就是让源代码更短小。

如果更短小的源代码是高级语言的目的，而某事物的力量在于它实现自己目的的程度，那么一门编程语言的力量就取决于它能把你的程序变多小。

反过来，一门不能让程序变小的语言，就没有做好编程语言应该做的事，就像一把切不快的刀，或者难以辨认的印刷品。

度量标准

不过，什么意义上的“小”呢？最常见的代码大小度量是代码行数。但我觉得这个指标之所以最常见，是因为它最容易测量。我不认为有人真的相信它是衡量程序长度的真正标准。不同语言对一行该写多少内容有不同惯例；在 C 语言里，很多行除了一个或两个分隔符之外什么都没有。

另一个简单的测试是程序中的字符数，但这个也不太好；有些语言（比如 Perl）只是比其他语言使用更短的标识符而已。

我认为更好的程序大小度量应该是元素的数量，这里“元素”指如果你画出一棵代表源代码的树，那么树上每个不同的节点就是一个元素。变量或函数的名字是一个元素；整数或浮点数是一个元素；一段字面文本是一个元素；模式中的一个元素或格式指令是一个元素；一个新的代码块是一个元素。虽然存在边界情况（-5 是两个元素还是一个？），但我认为对所有语言来说，大部分情况都是一致的，因此不影响相互比较。

这个指标还需要进一步完善，在具体语言上可能需要解释，但我觉得它试图衡量的正是正确的东西——即一个程序有多少个组成部分。我认为你在这次练习中画出的树，就是你在头脑中构想这个程序时必须构建的东西，因此它的大小与你编写或阅读它所需的工作量成正比。

设计

这种指标能让我们比较不同语言，但至少对我来说，这不是它最主要的价值。简洁性测试的主要价值在于作为设计语言时的指导。最有用的语言比较，是同一语言的两种潜在变体之间的比较。我能在语言里做什么来让程序更短？

如果程序的概念负担与其复杂度成正比，而一个程序员能承受的概念负担是固定的，那么这就等同于问：我能做什么来让程序员完成最多的事？这在我看来与“如何设计一门好语言”完全是同一个问题。

（顺便说一句，没有什么比设计语言更能明显证明“所有语言都是等价的”这个老生常谈是错误的了。当你在设计一门新语言时，你会不断比较两种语言——做了 x 的语言和没做的语言——来决定哪个更好。如果这真的是一个无意义的问题，那你干脆抛硬币就行了。）

以简洁为目标似乎是发现新想法的好方法。如果你能做一件事让许多不同程序都变得更短，那可能不是巧合：你很可能发现了一个有用的新抽象。你甚至可以写一个程序，通过搜索源代码中重复出现的模式来帮忙寻找。在其他语言中，那些以简洁著称的语言正是值得借鉴新想法的对象：Forth、Joy、Icon。

比较

据我所知，第一个写下这些问题的人是 Fred Brooks，在《人月神话》中。他写道，无论使用什么语言，程序员每天生成的代码量似乎都差不多。我二十出头第一次读到这句话时大吃一惊，觉得它有重大意义。这意味着（a）让软件写得更快唯一的方法是使用更简洁的语言，以及（b）一个肯花功夫这么做的人能把不这么做的竞争对手远远甩在身后。

Brooks 的假设如果成立，似乎正是黑客技术的核心。这些年来，我密切关注任何能得到的证据，从正式研究到单个项目的轶事。我没有看到任何能反驳他的东西。

我尚未看到在我看来决定性的证据，也不指望能看到。像 Lutz Prechelt 那样的编程语言比较研究，虽然产生了符合我预期的结果，但倾向于使用过于短小的问题，无法构成有意义的测试。一门语言更好的测试，是在一个月才能写完的程序中会发生什么。而如果像我一样相信语言的主要目的是便于思考（而不仅仅是在你想清楚后告诉计算机该做什么），那么唯一真正的测试就是你能在其中写出什么新东西。因此任何要求你满足预定义规格的语言比较，都在略微偏离正确方向。

一门语言的真正测试，是你能多好地发现和解决新问题，而不是你能多好地用它解决别人已经提出好的问题。这两个标准大不相同。在艺术中，刺绣和马赛克这样的媒介，如果你事先知道自己想做什么，就能做得很好；但如果你不知道，那就完全不行。当你想在创作过程中发现图像本身——比如画一个人像这么复杂的东西时必须这么做——你就需要使用更流动的媒介，比如铅笔、墨洗或油画。而实际上，挂毯和马赛克的制作方式是先画一幅画，然后照着复制。（“cartoon”这个词最初就是用来描述为此目的而画的草图。）

这意味着我们很可能永远不会有编程语言相对力量的准确比较。我们会有精确的比较，但不会有准确的。特别是那些明确为了比较语言而进行的研究，因为它们很可能使用小问题，而且必然使用预定义的问题，因此往往会低估更强大语言的力量。

来自实际项目的报告，虽然必然不如“科学”研究那么精确，却可能更有意义。例如爱立信的 Ulf Wiger 做过一项研究，认为 Erlang 比 C++ 简洁 4-10 倍，因此开发软件的速度也相应更快：

对爱立信内部开发项目的比较显示，包括软件开发所有阶段在内，每小时生成的代码行数大致相同，与使用的语言（Erlang、PLEX、C、C++ 或 Java）关系不大。那么区分不同语言的因素就变成了源代码量。

这项研究还明确讨论了 Brooks 书中只是隐含的一个观点（因为他测量的是调试后的代码行数）：用更强大语言编写的程序往往 bug 更少。这本身就成为一个目标，在网络交换机这样的应用中，可能比程序员生产力更重要。

品味测试

最终，我认为你必须跟着直觉走。在这门语言里编程的感觉如何？我认为找到（或设计）最好语言的方法，是让自己对一门语言让你思考的顺畅程度变得极度敏感，然后选择或设计感觉最好的那门语言。如果某个语言特性显得笨拙或限制性，别担心，你会感觉到的。

这种高度敏感是有代价的。你会发现自己无法忍受在笨拙的语言中编程。我觉得在没有宏的语言里编程限制得难以忍受，正如习惯了动态类型的人觉得必须回到每次都要声明变量类型、而且不能创建包含不同类型对象的列表的语言中编程，也难以忍受。

我不是唯一一个这样的人。我认识很多 Lisp 黑客都发生过这种情况。事实上，衡量编程语言相对力量最准确的指标，可能就是懂得这门语言的人中，有多大比例愿意接受任何能用到它的职位，而不管应用领域是什么。

限制性

我觉得大多数黑客都知道一门语言“感觉有限制”是什么意思。当你有这种感觉时，到底发生了什么？我认为这和你想走的街道被封锁、不得不绕很远路才能到达目的地时的感觉是一样的。你有话想说，而语言不让你说。

我认为这里真正发生的事是，一门有限制性的语言就是不够简洁的语言。问题不仅仅是你无法按计划表达想说的内容，而是语言强迫你绕的弯路更长。做一个思想实验。假设你想写某个程序，语言不允许你按计划的方式表达，却强迫你用另一种更短的方式来写。对我来说，这不会感觉很有限制性。这就像你想走的街道被封锁了，而十字路口的警察不是让你绕远路，而是直接给你指了一条捷径。太棒了！

我认为那种限制性的感觉有百分之九十来自被迫把你写出的程序变得比头脑中的版本更长。限制性主要就是缺乏简洁。所以当一门语言感觉有限制时，这（主要）意味着它不够简洁；而当一门语言不够简洁时，它就会让人感觉有限制。

可读性

我开头引用的那句话提到了另外两个品质：规范性和可读性。我不确定规范性是什么，也不确定规范且可读的代码比仅仅可读的代码有什么优势（如果有的话）。但我觉得我知道可读性指的是什么，而且我认为它也和简洁相关。

我们必须小心区分单行代码的可读性和整个程序的可读性。重要的是后者。我同意一行 Basic 代码通常比一行 Lisp 代码更易读。但用 Basic 写的程序会比用 Lisp 写的同一程序包含更多行（尤其是当你进入 Greenspunland 之后）。阅读 Basic 程序所需的总努力肯定更大。

总努力 = 每行努力 × 行数

我不像确信力量与简洁成正比那样确信可读性直接与简洁成正比，但简洁无疑是可读性的一个因子（数学意义上的因子，见上式）。因此，说一门语言的目标是可读性而不是简洁，可能甚至没有意义；这可能就像说目标是可读性，而不是可读性。

对第一次接触这门语言的用户来说，每行可读性意味着源代码看起来不会那么有威胁性。因此每行可读性可能是个好的营销决策，即使它是个糟糕的设计决策。这和“让人分期付款”这个非常成功的技巧是同构的：不是用高昂的前期价格吓到他们，而是告诉他们低廉的月供。然而分期付款对买家是净损失，而每行可读性对程序员可能也是。买家要付很多很多次那种低廉的月供；而程序员则要读很多很多那种单独来看可读的行。

这种权衡在编程语言出现之前就存在了。如果你习惯阅读小说和报纸文章，那么第一次读数学论文的经历可能会令人沮丧。读一页可能要花半小时。然而我相当确信问题不在记号法，尽管它可能给人这种感觉。数学论文难读是因为其中的思想难。如果用散文表达同样的思想（像数学家在发明简洁记号法之前不得不做的那样），它们并不会更容易阅读，因为论文会膨胀到一本书的长度。

究竟到什么程度？

不少人拒绝接受简洁等于力量的观点。我认为更有用的做法不是简单争论它们是否相同，而是问：简洁在多大程度上等于力量？因为很明显，简洁是高级语言的主要目的之一。如果不是全部目的，那么它们还有什么其他目的，而这些其他功能相对来说又有多重要？

我提出这个不是为了让辩论更文明。我真的想知道答案。一门语言在什么情况下（如果有的话）会简洁到对自己不利？

我开始时的假设是，除了病态例子，我认为简洁可以被视为与力量等同。我的意思是，在任何人会实际设计的语言中，它们是等同的，但如果有人想故意设计一门语言来反驳这个假设，他们大概能做到。其实我对这一点也不确定。

语言，而非程序

我们应该明确，我们讨论的是语言的简洁性，而不是单个程序的简洁性。单个程序当然可能写得过于密集。

我在 论 Lisp 中写过这个问题。一个复杂的宏必须节省远远超过自身长度的代码才能值得。如果写一个复杂的宏每次使用能为你节省十行代码，而宏本身是十行代码，那么只要使用次数超过一次，你就能获得净节省。但这仍然可能是个糟糕的选择，因为宏定义比普通代码更难阅读。你可能需要使用这个宏十次或二十次，才能在可读性上获得净提升。

我相信每门语言都有这样的权衡（虽然我怀疑随着语言变得更强大，代价也会更高）。每个程序员肯定都见过有人用可疑的编程技巧把代码变得稍微短一点。

所以在这点上没有争议——至少我没有。单个程序当然可能对自己不利地过于简洁。问题是，一门语言能这样吗？一门语言能否迫使程序员写出在元素层面很短、但整体可读性受损的代码？

很难想象一门语言会过于简洁的原因之一是，如果存在某种过度紧凑的表达方式，那么很可能也存在更长的表达方式。例如，如果你觉得大量使用宏或高阶函数的 Lisp 程序过于密集，如果你愿意，你可以写出与 Pascal 同构的代码。如果你不想在 Arc 中把阶乘表达为对高阶函数的调用

(rec zero 1 * 1-)

你也可以写出递归定义：

(rfn fact (x) (if (zero x) 1 (* x (fact (1- x)))))

虽然我一时想不出例子，但我对“一门语言是否可能过于简洁”这个问题很感兴趣。是否存在迫使你以一种僵硬且难以理解的方式写代码的语言？如果有人有例子，我会非常感兴趣。

（提醒：我寻找的是按照上面勾勒的“元素”指标非常密集的程序，而不仅仅是因为省略了分隔符且所有东西都用单字符命名而显得短小的程序。）
