有适用于 Outlook 的贝叶斯过滤器吗？

目前我知道的有九个：Spammunition、SpamBayes、Spam Bully、InboxShield、Junk-Out、Outclass、Disruptor OL、SpamTiger 和 JunkChief。

有没有什么办法可以保护我们公司的服务器？

最好的商用服务器级贝叶斯过滤器大概是 Death2Spam。而最优秀的开源贝叶斯过滤器之一 SpamProbe 也可以部署在服务器上运行。

大多数商用服务器级垃圾邮件过滤器仍然基于规则。不过已经开始出现一些采用贝叶斯过滤技术的产品。最简单的查找方法大概就是在 Google 上搜索。

你应该问销售人员的问题是：这个过滤器是否会根据我们收到的垃圾邮件和正常邮件来学习识别？如果它不学习，就不是贝叶斯过滤器。

Arc 或 Lisp 在编写这类软件时有什么优势吗？

Lisp 的符号（symbol）类型在处理单词数据库时非常有用，因为它允许通过比较指针来测试相等性。（如果你发现自己在用校验和来识别单词，其实你真正需要的是符号。）Prolog、Erlang 和 Ruby 也支持符号。Python 在一定程度上也支持，不过我认为它目前还没有专门的语法。

更广泛地说，Lisp 是为快速原型开发而设计的，而这个应用涉及大量这类工作。我大概把 95% 的开发时间都花在了 REPL 里，向其中输入表达式，在单个邮件上尝试各种调整后的算法。

我可以基于这个算法写过滤器吗？

当然可以。我并没有声称这里发明了什么新东西。贝叶斯文本分类是一个历史悠久且成熟的领域。如果说这篇文章里有什么新东西（至少对我来说是新的），那就是如此简单的一个变体在过滤垃圾邮件时竟然如此有效。

所以尽管去写垃圾邮件过滤器吧。这是一个非常有成就感的 Hack。如果你最终做出了别人可以使用的产品，请告诉我，我会加上链接。

我不懂 Lisp，你能给我解释一下这个算法吗？

可以用贝叶斯过滤来制作网页内容过滤器吗？

我没试过，但我觉得应该会很有效。

这个算法能过滤掉我 99.5% 的垃圾邮件且没有任何误判吗？

它确实过滤掉了我的 99.5% 的垃圾邮件。我在另外一个测试过的垃圾邮件集合上也得到了类似的结果。（我无法测量误判率，因为我只有垃圾邮件。）

我确实有点担心我的邮件可能特别容易过滤。我的很多垃圾邮件来自“opt-in”类型的群发者，比如 Qves 和 Virtumundo，而这类邮件非常容易检测。另外我自己的邮件里充满各种编程相关的词，这也让它很容易与垃圾邮件区分开来（不过讽刺的是，“programming”这个词本身经常出现在推销卫星天线的垃圾邮件里）。

所以如果你自己尝试实现这个过滤器，我很希望你能告诉我实际效果如何。我直觉上认为它对几乎所有人都会相当有效，但听到具体数据会更让人安心。

垃圾邮件发送者能不能通过在邮件里塞满随机单词来欺骗贝叶斯过滤器？

他们必须同时去掉坏词并加入中性词。只有最有信息量的 15 个词会影响最终概率，而中性词如“onion”不管出现多少次，都无法与罪证确凿的“viagra”在统计显著性上竞争。

为了压过罪证词，垃圾邮件发送者需要用特别“无辜”的词来稀释邮件，也就是那些不仅中性、而且在用户正常邮件中出现频率明显更高的词。但这些词（比如朋友的昵称、工作相关的术语）对每个收件人都不一样，而垃圾邮件发送者没有办法知道它们是什么。

一旦这个软件普及，垃圾邮件发送者难道不能调整他们的邮件来绕过它吗？

他们不一定能在保持原意的同时做到这一点。比如，如果他们想让你访问一个已被过滤器知晓的网址，就会很难调整绕过。

其次，用什么来调整？每个用户的过滤器都不一样，尤其是无辜词会特别不同。垃圾邮件发送者最多只能用纯粹中性的词来稀释邮件，而这些词通常不会出现在最有信息量的 15 个词之中，因此效果有限。

如果他们试图把大部分罪证词从消息中移除，就必须使用不同的委婉语，因为如果大家都用“adolescents”代替“teens”，那么“adolescents”很快就会获得很高的垃圾邮件概率。

最后，即使垃圾邮件发送者设法把所有罪证词都从正文中移除，这仍然不够，因为典型的垃圾邮件里大量罪证词都出现在邮件头中。

如果垃圾邮件发送者把邮件做成图片发送呢？

他们有时已经这么做了，而我们还是能抓住它们。这类邮件实际上包含大量确凿的内容。首先，邮件头还是和以前一样糟糕。而且别忘了我们不仅扫描纯文本，也会扫描 HTML。在正文里，除了图片通常还会有链接，两者都包含 URL，很可能会得到高分。“Href”和“img”本身的垃圾邮件概率都接近色情词汇。在 URL 中，图片文件名通常也远非随机。

你的程序能处理垃圾邮件发送者在单词中间插入 HTML 注释的伎俩吗？

可以，我在扫描 token 的层面就忽略 HTML 注释，甚至不把它们当作分隔符。

垃圾邮件发送者有时会使用随机生成的标签来打断 token，因为 HTML 渲染软件通常会忽略无意义的标签。我允许这些标签作为 token 的分隔符，而且效果很好。被打断的单词片段会直接获得很高的垃圾邮件概率。

这篇文章本身会被你的过滤器过滤掉吗？

不会。有人把它发给我做过测试，结果没有被过滤。虽然文章里包含很多垃圾邮件常用词，但也包含更多明显无辜的词（讽刺的是，其中包括“spammers”这个词，它在我真实的邮件里经常出现，而在垃圾邮件里从未出现过）。由于只有最有信息量的词才起作用，那些无辜词在排序中挤掉了垃圾词，最终这封邮件得到了可能的最低垃圾概率。

这很合理，因为一个人写的文章，比起垃圾邮件来，更可能和他自己的邮件内容相似，即使这篇文章恰好是关于垃圾邮件的。

你的软件过滤关于垃圾邮件的邮件效果如何？

这是所有做垃圾邮件过滤的人都会遇到的问题。在极端情况下，如果某个从未联系过的人只是单纯转发了一封垃圾邮件而没有附加任何评论，那确实很难过滤。这时就完全取决于邮件头了。

但话说回来，如果有人能毫无顾忌地转发垃圾邮件给你，那么垃圾邮件发送者也可以这么做。所以对于这类情况，我们可能需要一个特殊的密码，让人们可以在邮件中包含它以绕过过滤器；同时（我已经有一个了）需要一个专门存放“元垃圾邮件”的垃圾文件夹，这样它们就不会影响概率计算。

如果说这事有什么正面意义，那就是：如果我们能做出对自己足够好用的过滤器，那么它们对其他人会更好用。

这个软件会过滤掉正常的自动化邮件吗，比如订单确认和新闻组 FAQ？

大多数情况下不会。那类邮件可能是自动生成的，但文本特征通常和垃圾邮件有很大不同。

唯一容易被过滤的自动化回复是那些包含大量额外广告的。本质上，这些邮件就是一个自动化回复后面附加了一段垃圾邮件，所以过滤器会捕捉到它们并不奇怪。

那些会回复垃圾邮件的人会使用过滤器吗？

我认为是会的。这只是我的猜测，但我猜那些 会回复垃圾邮件的人，通常是通过 Yahoo Mail、Hotmail 之类的免费服务，或者 AOL、Earthlink 这样的大型 ISP 来收邮件。一旦“可以过滤掉大部分垃圾邮件”的消息传开，他们就会被迫提供有效的过滤功能。过滤器会消灭垃圾邮件吗？

如果过滤器能拦住大部分垃圾邮件，垃圾邮件发送者会不会 просто发更多来弥补？

垃圾邮件发送者已经是在满负荷运转了。

Bayes' Rule 不是假设各个预测因子相互独立吗？

是的，严格来说，只有假设例如“sex”出现在邮件中的概率与“sexy”出现在其中的概率无关时，它才完全成立。显然实际情况并非如此。

另一方面，违背这一假设的做法有着悠久的传统。这样做被称为“naive Bayesian”（朴素贝叶斯）算法，而在实践中它的效果相当好，就像在实践中（只要远离精度边界）我们可以把浮点数当作实数来用一样。

这个算法会考虑一封邮件是垃圾邮件的先验概率吗？

不会。它假设一封邮件是垃圾邮件或非垃圾邮件的概率相等。因此，如果能考虑每个用户实际收到的垃圾邮件比例，准确率可以略有提升。不过必须按小时来做这件事，因为不同时段的比例差异极大，以至于整体先验概率作为预测因子毫无用处。

你为什么选择 15 个词？15 这个数字有什么特别之处？

实际上，在最新版本的过滤器中我使用了 20 个词，因为通过 来标记 token 后，软件现在每封邮件能看到更多不同的 token。context

几乎所有垃圾邮件都会包含这么多相当有罪证性的词。如果你看得更多，就会开始引入中性词，而中性词充其量只是噪音。但你又希望看得尽可能多，因为合法邮件也可能包含两三个具有高垃圾概率的词——比如如果你只用前 5 个词，就很容易产生误判。

这个方法对英语以外的其他语言有效吗？

只要它们使用空白字符来分隔 token，应该就可以。
