Arc 的一些工作

October 2003 · 阅读原文

2003 年 10 月

（这是受邀在 ILC 2003 大会上发表的演讲。）

几天前，当我正在准备这篇演讲时，slashdot 有一篇链接指向我之前写的关于垃圾邮件的文章。通常遇到这种情况，我会尽量避免去看下面的评论。因为读到有用内容的概率远低于被惹恼的概率。但这次我想偷个懒，就往下翻了翻，第一眼看到的一篇帖子标题是“回去工作吧，Paul”，说我不应该把时间浪费在写垃圾邮件过滤器上，而是应该回去继续做 Arc。

其实我并没有像这位老兄想的那样偷懒。我经常收到人们的邮件，本质上都在问：“我们什么时候才能用上？”简短的回答是：还没有。更完整的回答是，这个项目目前处于四个阶段中的第三个。最初的计划是：(a) 先做出一个简陋的 Arc 初始版本，(b) 在实际应用中用上一段时间，然后 (c) 回过头来写一份完整、干净的语言规范，最后 (d) 以此为基础实现一个相当不错的版本。

我现在处于 (c) 阶段。我不知道完成规范还需要多长时间。事实证明这相当困难，但也非常有趣。既然我目前主要在做规范，那今天就主要谈谈这个。

我认为一门编程语言应该主要由它自身来定义。语言规范应该分成两部分：一小部分充当公理的核心运算符，以及语言的其余部分——就像定理一样，用这些公理来定义。

这样设计语言有两个主要原因。一是能产生更加优雅的语言。二是通过这种方式，你可以让语言具有最大的可重写性。任何用这门语言写的东西，都可以被使用这门语言的程序员重写。

允许人们重写你的语言是个好主意。作为语言设计者，你不可能预见到程序员将来想用它做的所有事情。在他们能够重写语言的范围内，你就不必自己去做了。

但可重写语言的好处不止于让程序员修正你的错误。我认为最优秀的程序员往往是通过重写他们正在使用的语言来进行工作的。因此，即使存在所谓完美的语言，它也应该是高度可重写的。事实上，如果要我猜，我觉得最完美的语言可能就是可重写性最强的那一个。

将编程语言公理化的想法当然不是什么新点子。它几乎和编程语言本身一样古老。在 1960 年那篇著名的论文中，Lisp 的根源展示了如何通过定义一门他称为 Lisp 的语言来做到这一点。你可能对它很熟悉。如果你有七个基本运算符（quote、atom、eq、car、cdr、cons 和 cond），就可以用它们定义另一个函数 eval，它的作用就是一个 Lisp 解释器。

因此，任何包含这七个运算符的语言都是 Lisp 的一个方言，无论它原本是否打算如此。这对任何正在设计新语言的人来说一定是个令人警醒的前景。这七个公理反应极其强烈，只要把它们放在一起，它们就会爆炸，结果你设计的不是一门新语言，而是一门老语言。所以如果你想设计一门新的编程语言，关键在于不要让它太过强大。

最近我一直试图从 McCarthy 1960 年的论文中断的地方继续下去。我长期以来都怀疑，Lisp 之所以成为如此优秀的编程语言，主要原因在于它并不是被设计成一门编程语言的。它实际上是一个理论结果。如果你尝试回答这个问题：要写一个能自我解释的语言解释器，最少需要多少个运算符？得到的答案就是 Lisp。（至少，它是其中一个答案；这个问题并不十分精确，所以可能存在多个答案。）

换句话说，Lisp 与其说是 McCarthy 发明的东西，不如说是他发现的东西。这似乎是一门编程语言应该具备的良好品质。我在观察 C 和 Smalltalk 时，也会产生某种同样的必然性感觉。

当然，McCarthy 的规范一落到黑客手里，所有这些理论探讨就立刻被打断了。在 Lisp 1.5 中，read 和 print 并没有用 Lisp 本身来编写。考虑到当时可用的硬件，这也是不可能的。但现在情况不同了。借助当今的硬件，你可以一直推进，直到为一个完整的编程语言写出一个可运行的规范。这就是我一直在做的事情。

我目前试图回答的问题是：要在最初的七个运算符基础上再添加哪些运算符，才能写出一个能解释完整编程语言的 eval？

当然，这取决于你对“完整编程语言”的定义。但我认为有一些特性是大家都会认同的必需品。例如，我们需要 I/O，否则程序甚至无法进入计算机被求值。我们需要某种处理错误的方案。（McCarthy 最初的 eval 假设它的参数是一个正确的程序。如果你引用一个未绑定的变量，它会进入无限循环。）而且我们需要比符号和 cons 更多的基本数据类型。比如，我们很可能需要数字。

这项工作我还没有完成，但到目前为止，我惊讶地发现，为了让这些特性工作起来，你需要在核心里添加的原语竟然如此之少。我认为要定义新类型，只需要三个新的原语（加上赋值和词法作用域）就够了。其中一个新原语会替换原来的 atom，所以最终总数仍然只有九个。

在 McCarthy 最初的 eval 中，仅有的数据类型是 cons 和符号。原则上，你大概可以用 cons 来表示任何你想要的东西。例如，整数 n 可以用长度为 n 的列表来表示。

当然，这在实践中会极其低效。也没有人提议实现真的这么做。写 eval 的目的是定义语言规范，而不是语言实现。在内部，实现可以做任何它想做的事——包括用最适合硬件的方式来表示数字——只要它们对外表现出的行为与作为语言规范的解释器无法区分即可。

用列表表示数字的真正问题不是低效，而是如果我们这么做，就无法区分数字和列表。有一个函数会希望区分它们，那就是 print 函数，它需要把一个包含三个元素的列表打印成 (a a a)，而把数字 3 打印成 3。

因此我们需要某种数据类型的概念。而且如果可能，我们应该像 Lisp 的其他部分一样，让用户也能使用这种机制。正如用户可以定义新的函数，这些函数和预定义函数完全一样，用户也应该能够定义新的类型，这些类型和预定义类型完全一样。当然，我们希望把尽量少的东西放进核心。例如，复数就不应该必须在语言核心中定义。

我们能做的最少的事情是什么？看起来定义三个新的基本函数就足够了，我把它们叫做 tag、type 和 rep。[1]

tag 接受两个参数：一个类型标签和一个表示。因此你可以创建一个类型为符号 a、表示为符号 b 的对象，像这样：

(tag 'a 'b)

另外两个运算符 type 和 rep 则用来拆解这样的对象。

(type (tag x y)) -> x

(rep (tag x y)) -> y

我预期类型名通常会是符号，但不一定是。两个参数都可以是任意类型。我想象不出用户为什么会想要使用符号以外的类型标签，但我也看不出有什么理由要阻止他们这么做。

也许现在是个好时机，来谈谈我在 Arc 中对待限制的态度。语言设计者禁止某些事情似乎有三个原因：因为不存在一致的方式允许它，因为它难以高效实现，以及因为允许它可能会让程序员陷入麻烦。

第一类限制的例子是不允许程序对符号调用 car。这种请求在语义上是形式错误的，就像问“蓝色有多重”一样。（我在大学学哲学时学到的少数东西之一就是，大多数传统哲学争论同样是形式错误的。自由意志甚至个人身份之类的概念都经不起仔细推敲。）你必须禁止用户提出形式错误的问题，因为根本无法回答它们。

但我不会采用另外两种限制。在 Arc 中，提升效率的方案不是去限制语言规范，而是允许用户在需要更高速度的地方，通过声明有选择地限制自己的程序。这和那个著名的错误（如果它真的是错误的话）——假设存在“足够聪明的编译器”——不是一回事。我不是说让编译器自动找出哪里可以偷懒，而是让用户在良好性能分析工具的帮助下，告诉编译器哪里可以偷懒。

例如，Arc 目前有一等宏。这似乎是定义语言最简单的方式。一等宏有可能非常低效；你可能会在运行时展开宏调用。但我认为大多数用户不会想要利用这种可能性。他们通常会做一个全局声明，说明所有宏调用都可以在编译时展开，这样宏的开销就不会比现在的 Lisp 更高。

第三类限制的例子是封装，这类限制的目的纯粹是为了防止用户陷入麻烦。如果我能决定，Arc 中不会有这类限制。可能（也可能不会）存在某些情况，让这种限制性的语言成为净赢家，但那不是我想要写的语言类型。我的目标是小规范和小程序，而这类限制两者都给不了你。

我说“如果我能决定”，是因为我发现设计语言就像其他形式的绝对权力一样，会绝对地腐蚀人。即使在多年强调“语言应该是仆人而不是主人”之后，我仍然发现自己会想：我应该让用户做某某事情吗？我认为唯一的防御手段是定下一条规则：如果你发现自己开始问“是否应该让用户……”，就要自动回答“是的，只要没有逻辑上的理由禁止它”。

因此，例如，在 Arc 中，在参数列表里使用同一个变量两次并不是非法的。对这样的代码存在一种一致的解释——从左到右绑定参数——我就是这么做的。（在 McCarthy 最初的 eval 中，使用同一个参数两次也能工作，只是第一个参数会成为可见的那个。）也许大多数出现同一个符号两次的参数列表都是 bug 导致的。但也不能排除某些自动生成的代码（比如宏展开）可能有意想这么做。更重要的是，这类 bug 应该由开发环境中的某种类似 lint 的组件来捕捉。它不应该是语言核心的工作。

我们还需要对这三个新原语再做一些具体规定。如果你对一个已经是给定类型的对象再次调用 tag，你只会得到第二个参数。所以

(tag 'symbol 'a) -> a

type 函数对 cons 返回 cons，对符号返回 symbol，对函数返回 fn。

(type cons) -> fn

rep 函数在应用于符号、cons 或基本函数时，直接返回其参数。

(rep 'a) -> a

最后，当你把一个列表作为 tag 的第二个参数时，对结果对象调用 rep 将返回完全相同的那个列表。

(let x '(a b c) (is (rep (tag 'foo x)) x)) -> t

由于 Arc 有赋值，这意味着用户可以破坏性地修改对象的表示。这可能是个愚蠢的行为，但谁知道呢。既然没有纯粹逻辑上的理由禁止它，那我就不禁止。

据我所知，这就是让新类型工作起来所需的核心全部内容。如果你想重载现有运算符，以便在传入你的新类型时做正确的事情，你不需要在核心里增加任何新东西。只要你有词法作用域，就可以用一个新的运算符定义包裹旧的定义。因此，如果你想修改 print，让它以特殊方式显示你的新类型 foo 的对象，可以这样写：

(let orig print (def print (x str) (if (is (type x) 'foo) ...new code... (orig x str))))

通过暴露 eval 的几个子过程，我成功地连宏都不必放进核心。这里是我目前对宏的定义：

(def macex (op args) (apply (rep op) args idfn))

(let orig evexpr (def evexpr (op args env cont) (if (is (type op) 'mac) (eval (macex op args) env cont) (orig op args env cont))))

(let orig expand= (def expand= (place val env) (eval (car place) env (fn (op) (if (is (type op) 'mac) (expand= (macex op (cdr place)) val env) (orig place val env))))))

evexpr 和 expand= 这两个函数分别是求值表达式和生成赋值展开的函数。宏只是一个被附加了类型 mac 的函数。例如，下面是 def 的定义：

(= def (tag 'mac (fn (name parms . body) (list '= name (cons 'fn (cons parms body))))))

而下面是 mac，它相当于 Arc 版的 defmacro：

(= mac (tag 'mac (fn (name parms . body) (list '= name (list 'tag ''mac (cons 'fn (cons parms body)))))))

用它我们可以这样定义 let：

(mac let (var val . body) (list (cons 'fn (cons (list var) body)) val))

除了那些按定义无法用语言本身编写的原语运算符之外，整个 Arc 规范都将用 Arc 自身来编写。正如 Abelson 和 Sussman 所说，程序必须是为了让人阅读而写，只是附带地为了让机器执行。因此，如果一门语言足够好，那么源代码就应该是一种比英文散文更好的传达思想的方式。[2]

这种方法的一个结果是，你可能在不知不觉中就设计出了某些特性（或 bug）。如果你拿出一段代码说，这就是语言定义，那么它可能像任何程序一样，做出（在这个例子里，也就是意味着）一些你并未有意为之的事情。

我认为我们不应该对此感到惊慌。数学中也是如此。当某个数学家描述一类事物时，他并不需要知道它的所有性质，就能判断它是否是一类值得讨论的有用事物。同样，我认为如果我们设计的语言，其规范是一个看起来正确的程序——一个简短且没有权宜之计的程序——那么它很可能是一门好语言，即使我们并不完全确定它具体做了什么。

我并不自称知道我正在写的 Arc 规范的所有后果，就像 McCarthy 也不可能知道他最初 Lisp 定义的所有后果一样。但至少，如果基本运算符的行为被完全明确规定了，那么它就是无歧义的。这一点 Common Lisp 肯定做不到。当一个 Common Lisp 宏返回一个 car 部分是函数（注意，不是函数名，而是一个真正的函数对象）的列表时会发生什么？在我用过的所有实现里，发生的事情都是你所期待的。但规范对此只字未提。至于 loop 宏，ANSI 标准作为教程都勉强，更别说作为定义了。

说到这个，我怀疑用代码作为规范的另一个好处是它能让你保持诚实。如果你能看到定义 loop 所需的代码，它那么庞大而复杂，显然就能看出哪里出了问题。如果每个人都必须裸体出门，我们大概都会更健康。同样，如果语言定义像它们的实现一样是开源的，语言大概也会更干净。

我相信，这三个新的类型运算符，再加上包装函数的技术，给了我们比人们通常所说的“面向对象编程”更一般的东西。我们可以把一个函数包装在针对参数任意属性的“特化”代码中，而不仅仅是它们是否属于某些类型，或类型的并集（即超类是什么）。

如果你想定义一种更具体的、绑定到继承机制的重载形式，我认为那不会太难。目前重要的一点是，这不是你需要在语言核心里考虑的事情。如果你想定义一门面向对象的语言（无论你具体指什么），看起来你在核心里需要的无非就是这三个类型原语，以及用新定义包装旧函数的技术。

当我意识到采用公理化方法能给我一个正当理由，不必在 Arc 核心里塞满面向对象特性时，我大大松了一口气。我之前一直隐隐担心，也许 Arc 应该成为那种“一切都是对象”的现代语言，或者“对象一层层到底”的语言。说实话，我并没有很担心这个，但似乎有大约 1% 的可能性，我不得不这么做。

如果有人抱怨 Arc 的面向对象特性不够，我可以拿公理化的严格要求来辩解。对不起，但我被限制只能在核心里放入最少可能的功能。当然，我个人的猜测是，这种最少的功能其实就是你在大多数时候真正想要的……

我现在相当有信心已经解决了这个问题。在 Arc 里，“一切都是对象”。但对象只是任何具有类型的东西。你可以询问一个对象的类型，也可以重定义任何运算符，让它在遇到特定类型的对象时做特殊处理。因此，如果我想构建一个基于参数类型进行函数重载的复杂系统，我可以把它做成一个库。不过，不知怎的，我觉得我可能永远都不会去实现它。

我还在研究的另一件事是错误处理。和运算符一样，我希望识别尽量少的错误。因此这项练习最终会告诉我，最小的错误集合是什么，同时也告诉我最小的原语运算符集合是什么。

类型也是如此。到目前为止，仅有的基本类型是符号、cons 和函数。我将不得不加入流（stream），但除此之外可能不需要再加很多。数值类型都将用列表来定义。我不确定是否要加入一个位（bit）类型，还是直接使用符号列表。

我也许能避免引入一个单独的字符类型，让 Arc 版的 read-char 直接返回单字母的符号。我承认这听起来是个奇怪的想法，但目前我还想不出不这么做的理由。

不过，这整个事业有一个特点：它令人惊讶。我不断发现，有些我想做的事情要么比我预想的难得多，要么容易得多。对我来说这是个好兆头，因为它意味着我正处在一片相对未被探索的领域。

但当然，事情随时可能灾难性地出错。我还有很多工作要做才能完成 Arc 规范。

在此期间，任何因为 Arc 似乎花了太长时间还没出来而感到沮丧的人，或许可以考虑一下如何在语言核心之外实现可选参数，同时在求值默认表达式期间对续延（continuation）做正确的事情——不管那正确的事情是什么——这正是我停下来写这篇演讲时正在做的工作。我认为我能做到，但我必须先弄清楚我到底想做什么，才能判断它是否可能。这就是炮制一个语言规范的感觉。
