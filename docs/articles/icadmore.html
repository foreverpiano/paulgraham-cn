再谈《书呆子的复仇》

阅读原文

书呆子的复仇 引发了许多后续讨论，我在此为有兴趣深入探讨其中议题的朋友们做个总结。

Trevor Blackwell 写道，Lisp 只在某些类型的项目中才能发挥优势。他的邮件非常清晰，我直接原文引用如下：

我觉得如果你能限定 Lisp 真正优越的适用范围，说服力会更强。比如在以下领域它可能并不更好：

- 通信、界面、格式转换 - 实时控制 - 数值密集型代码 - 操作系统实现 - 快速处理文件的简短脚本

而以上这些其实占了所有软件的很大比例。

Lisp 更适合信息处理：在复杂数据集上运行非常复杂的算法。只是在最近的万维网时代，人们才开始大量做这类工作。

在 Web 兴起之前，我认为只有极少数软件包含复杂算法：排序大概就是最复杂的了。你能在 FreeBSD 里找到什么复杂算法吗？Nortel 交换机那五千万行代码里可能连个排序都没有。全都是在管理硬件和资源，以及如何优雅地处理故障。Cisco 路由器也有几百万行软件，但其中复杂的算法并不多。

ITA 所写的那类软件其实非常罕见。十年前，真正包含复杂算法的软件大概只有 CAD 设计与综合，而它们确实在使用 Lisp。

说软件类型的转变让 Lisp 重新成为优秀选择，比说过去四十年里绝大多数程序员都是笨蛋要更可信。

我同意，如果你想做真正底层的操作，比如搬动比特，而且处理器时间比程序员时间更宝贵，那么 Lisp 可能不是合适的语言。那种情况下你会想要用 C 甚至汇编语言。但我认为这类应用很少，而且越来越少（参见摩尔定律）。我知道有好几个地方在用 Lisp 开发实时应用，包括 Rod Brooks 的机器人研发实验室。

我也同意，如果要写与 C 语言程序紧密配合的代码，Lisp 可能不是理想选择，但这不是 Lisp 的错。同样，C 也不是用来写与 Lisp 紧密配合程序的理想语言。

我不同意说 Lisp 不适合写快速小脚本。它交互式的特性恰恰让它特别擅长这类工作。

我也不同意“绝大多数程序员四十年来都是笨蛋”这个说法不可信。在我看来这完全有可能。如果只看用户数量，当前任何技术领域（实际上几乎任何领域）的领导者大多是平庸的。看看 Windows 就知道了。

技术创新通常要花几十年才能普及。大众汽车在 1930 年代就开始生产承载式车身汽车。到了 1980 年代，几乎所有新车都采用这种设计。难道只是因为绝大多数汽车设计师四十年来都是笨蛋吗？没错，虽然我觉得人们更喜欢用“保守”这个委婉说法。

Bengt Richter 提出了另一个 Python 解决方案：

def foo(n): def bar(i): bar.s += i return bar.s bar.s = n return bar

但我认为 Python 程序员仍然把定义一个新类视为标准解决方案。

好几位 Python 用户写信告诉我，你无法用 Python 写出 Lisp/Perl/Smalltalk/Javascript 那段程序的原因是 Python 的词法变量不可变，而且“匿名函数”只能包含一个表达式。当我问为什么不能直接写

def foo(n): lambda i: n += i

的时候，我不是在问当前 Python 定义中是什么阻止了这种写法。我知道 Python 目前强加了这些限制。我想问的是这些限制给你带来了什么好处？如果无法修改外层作用域的变量值，或者无法在 lambda 中放入多个表达式，这如何让 Python 成为更好的语言？区分表达式和语句又给你带来了什么？

对匿名函数能写什么内容的限制尤其站不住脚。事实上，“匿名函数”这个说法本身就暗示了思维的局限性。你会把一个没赋给变量的哈希表叫做“匿名哈希表”吗？如果函数是你语言中的一种数据类型，就应该像对待其他数据类型一样对待它们。一个具名函数应该只是碰巧赋给了一个变量的函数。限制一个未赋给变量的函数能做什么，跟限制一个未赋给变量的哈希表或字符串能做什么，道理是一样的荒谬。

Thomas Herchenroeder 写信强调，语言必须适合普通程序员，而不仅仅是超级黑客：

我认为你设想的是一个优秀黑客，他的代码偶尔会被其他优秀黑客阅读，他们惊叹于他精妙的算法和编码风格。他们有兴趣学习新东西，也渴望理解一切，纯粹为了智识上的挑战。不幸的是，那不是我（以及很多其他人）所生活的世界。我的同事……希望能以最小的努力就能看懂我的代码。因此一门语言的威力就和普通 IT 专业人员能轻松消化什么内容联系在一起，而后者又是由大学课程中普遍教授、在容易获得的书籍中解释、以及在流行文章中讨论的内容所决定的。归根结底就是关于编程、语言、算法和模式的主流知识。在某种程度上，你必须随大流。再说 Python：它以简洁优雅的方式实现了主流概念。

所以，也许我们又回到了原点：我们需要一种给超级用户、黑客使用的“强大”语言，以及另一种给日常公司里的普通人使用的“强大”（意义不同）语言。

我同意，为新手和动力不足的程序员设计的语言有其存在的价值。我唯一不同意的是，是否应该用“强大”这个词来描述这种特性。我认为描述你提出的“强大”的第二种含义，已经有一个词了，那就是“易用”。法拉利是强大的。大众高尔夫是易用的。

当已经有一个现成的词能准确表达你想说的特性时，为什么还要扭曲语言，去寻找某种比喻意义来称高尔夫是强大的呢？而且，如果你把“强大”这个词用得太宽泛而模糊了它的含义，你就再也没有一个词来指代法拉利所拥有的那种非常明确的特性了。

我认为至少我们中的一些人，保持关注法拉利意义上的“强大”是很重要的。必须有人这么做，因为那是下一代“主流知识”的源头。1980 年的时候，如果大学课程和容易获得的书籍谈到了结构化编程，就已经算是很先进的了。现在它们会讨论垃圾回收和动态类型这些话题，而在 1980 年这些还会被认为是 esoteric 的东西。

Paul Prescod 说我故意选了一个对 Lisp 有利的例子。如果我想那么做，我会写一个宏而不是一个函数。如果说我选的例子有偏向性，那也是对 Lisp/Perl/Smalltalk/Javascript 有利，因为这段代码在这四种语言里同样简洁（以及很多累加器生成器）。

实际上我对 Python 的表现之差感到惊讶。比如我从来没意识到 Python 的 lambda 表达式不能包含和具名函数一样的内容，也不知道外层作用域的变量可见但不可修改。无论是 Lisp、Perl、Smalltalk 还是 Javascript 都没有施加这两种限制。

我看不出这两种限制带来了什么好处。我能明白 Python 逐步的、持续的（= 不完整的）演化过程如何产生了它们。所以奥卡姆剃刀原理告诉我们，后者才是 Python 成为这样的原因。也就是说，这些限制是 bug，而不是特性。

Paul 已经写了一篇针对《书呆子的复仇》的反驳：《Python 与 Lisp 的关系》（Peter Norvig 也写过相关主题的文章：《给 Lisp 程序员看的 Python》）。

在 LL1 上有大量讨论，讨论是否像我在《书呆子的复仇》附录中暗示的那样，简洁性应该是编程语言设计的一个良好目标。这个问题在简洁就是力量中单独讨论。
