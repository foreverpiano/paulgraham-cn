设计与研究

January 2003 · 阅读原文

2003 年 1 月

（本文改编自 2002 年秋季 NEPLS 会议的主题演讲。）

来到这个国家的访客常常惊讶地发现，美国人喜欢以“你是做什么的？”作为谈话的开场白。我一直不喜欢这个问题，因为我很少能给出简洁的答案。但我认为自己终于解决了这个难题。现在，当有人问我做什么时，我会直视对方的眼睛说：“我在设计一种新的 Lisp 方言。”我向所有不喜欢被问及职业的人推荐这个回答。谈话会立刻转向其他话题。Arc

我并不认为自己在进行编程语言的研究。我只是在设计一门语言，就像有人设计一栋建筑、一把椅子或一种新字体一样。我并不是在尝试发现什么新东西。我只是想创造一门适合编程的优秀语言。在某些方面，这种定位让事情变得容易得多。

设计与研究的区别似乎在于“新”与“好”之间的权衡。设计不必是新的，但必须是好的。研究不必是好的，但必须是新的。我认为这两条路径在顶端会汇聚：最好的设计通过运用新思想超越前人，而最好的研究则解决那些不仅新颖、而且真正值得解决的问题。因此，我们最终的目标是一致的，只是从不同方向接近它而已。

我今天要讨论的是，从背后看你的目标是什么。当你把编程语言当作一个设计问题而非研究课题来对待时，你会采取哪些不同的做法？

最大的区别在于，你会更加关注用户。设计始于询问：这是为谁设计的，他们需要从中得到什么？举例来说，一位优秀的建筑师不会一开始就创造一个设计然后强加给用户，而是先研究预期的用户，并弄清楚他们真正需要什么。

请注意我说的是“他们需要什么”，而不是“他们想要什么”。我不想给人留下这样的印象：作为设计师工作就意味着像快餐厨师一样，按照客户吩咐的去做。在艺术领域，这种情况因领域而异，但我认为没有任何一个领域的最优秀作品是由那些仅仅严格按照客户要求去做的人创造的。

“客户永远是对的”这句话，在“好设计以对用户有多大帮助来衡量”这个意义上成立。如果你写出一本让所有人都感到无聊的小说，或者设计出一把坐起来极不舒服的椅子，那么你的工作就是失败的，仅此而已。辩解说这部小说或这把椅子是按照最先进的理论原则设计的，根本不能成为借口。

然而，做出对用户有用的东西，并不意味着简单地按照用户告诉你的去做。用户并不知道所有可能的选择，而且常常对自己真正想要的东西判断错误。

我认为这个悖论的解答是：你必须为用户进行设计，但你要设计的是用户真正需要的东西，而不仅仅是他声称自己想要的东西。这很像当医生。你不能只治疗患者的症状。当患者告诉你他的症状时，你必须找出他真正的问题所在，并针对那个问题进行治疗。

这种对用户的关注是一种公理，从中可以推导出大多数良好设计的实践，而且大多数设计问题也都围绕它展开。

如果好的设计必须满足用户的需求，那么谁是用户呢？当我说设计必须为用户服务时，并不是暗示好的设计要以某种最低公分母为目标。你可以选择任何你想要的用户群体。例如，如果你正在设计一个工具，你可以为从初学者到专家的任何人设计，而对某一群体来说好的设计，对另一群体可能就是糟糕的。关键在于，你必须选择某个用户群体。我认为，如果不参照某个预期的用户群体，你甚至无法谈论什么是好设计或坏设计。

如果预期的用户包括设计师本人，你最有可能做出好的设计。当你为一个不包括你自己的群体设计东西时，往往是为那些你认为不如你 sophistication 的人设计的，而不是更 sophistication 的人。

这是一个问题，因为无论多么善意地俯视用户，似乎都不可避免地会腐蚀设计师。我怀疑美国很少有住宅项目是由那些预期自己会住在里面的建筑师设计的。你可以在编程语言中看到同样的现象。C、Lisp 和 Smalltalk 是为其自身设计者使用的而创造的。Cobol、Ada 和 Java 则是为其他人使用的而创造的。

如果你认为自己在为傻瓜设计东西，那么很可能你设计出来的东西即使对傻瓜来说也不够好。

不过，即使你是在为最老练的用户设计，你仍然是在为人类设计。这与研究不同。在数学中，你选择抽象不是因为它们对人类来说容易理解，而是选择那些能让证明更简短的。我认为这在一般科学领域也是成立的。科学思想并不是为了符合人体工程学而存在的。

在艺术领域，情况则大不相同。设计完全是关于人的。人体是一种奇怪的东西，但当你设计一把椅子时，你就是在为它设计，而且没有办法绕过去。所有的艺术都必须迎合人类的兴趣和局限。例如，在绘画中，在其他条件相同的情况下，包含人物的画作会比没有人物的更有趣。文艺复兴时期的伟大画作全都充满人物，这并非仅仅是一个历史偶然。如果不是这样，绘画作为一种媒介就不会拥有它今天所拥有的声望。

不管你喜不喜欢，编程语言也是为人而存在的，我怀疑人类大脑和人体一样，也是块状且特立独行的。有些想法对人来说容易掌握，有些则不然。例如，我们处理细节的能力似乎非常有限。正是这一事实使编程语言从一开始就成为了一个好主意；如果我们能够处理所有细节，我们就可以直接用机器语言编程了。

还要记住，语言主要不是用来呈现最终程序的形式，而是程序必须在其中被开发出来的东西。任何从事艺术的人都会告诉你，在这两种情况下，你可能需要不同的媒介。例如，大理石是一种适合表达已完成思想的美好、耐久的媒介，但对于发展新思想来说，却是一种毫无灵活性的媒介。

一个程序，就像一个证明，是对一棵树的修剪版本，这棵树过去曾有过四处分支的错误开端。因此，对一门语言的检验不仅仅是看最终程序在其中看起来有多干净，而是看通往最终程序的路径有多干净。一个能让你得到优雅最终程序的设计选择，可能不会给你一个优雅的设计过程。例如，我写过几个充满嵌套反引号的宏定义宏，现在看起来像小宝石一样，但写它们花了好几个小时最丑陋的试错过程，而且坦白说，我仍然不能完全确定它们是正确的。

我们常常表现得好像检验一门语言的标准就是看最终程序在其中看起来有多好。当你看到同一个程序用两种语言写出来，其中一个版本明显更短时，这看起来非常有说服力。当你从艺术的角度接近这个问题时，你就不太可能依赖这种测试。你不会想最终得到一门像大理石一样的编程语言。

例如，在软件开发中，拥有一个交互式的顶层环境（在 Lisp 中称为 read-eval-print 循环）是一个巨大的优势。当你拥有它时，这会对语言的设计产生实际影响。例如，在一门要求在使用变量前必须先声明它的语言中，这就无法很好地工作。当你只是在顶层环境中输入表达式时，你希望能够把 x 设置为某个值，然后开始对 x 进行各种操作。你不希望必须先声明 x 的类型。你可以质疑这两个前提中的任何一个，但如果一门语言为了方便必须拥有顶层环境，而强制类型声明与顶层环境不兼容，那么任何强制要求类型声明的语言都不可能是方便编程的。

在实践中，要做出好的设计，你必须接近用户，并持续接近用户。你必须不断地用真实用户来校准你的想法，尤其是在开始阶段。简·奥斯汀的小说如此优秀的原因之一，是她会把它们大声朗读给家人听。这就是为什么她从未陷入自我沉溺的艺术化风景描写，或是自命不凡的哲学思辨。（哲学是存在的，但它被编织进了故事之中，而不是像标签一样贴在上面。）如果你打开一本普通的“文学”小说，想象把它当作你自己写的东西大声读给朋友听，你就会痛切地感受到那种东西对读者是多么大的强加。

在软件世界中，这个想法被称为“Worse is Better”。实际上，“Worse is Better”这个概念中混合了几种不同的思想，这也是为什么人们还在争论“更差是否真的更好”。但其中一个主要思想是：如果你正在构建新的东西，你应该尽快把原型放到用户面前。

另一种方法可以称为“孤注一掷策略”。不是快速推出原型并逐步改进它，而是试图一次性长传达阵完成一个完整的、最终的产品。据我所知，这是一条通往灾难的道路。无数初创公司在互联网泡沫期间就是这样毁灭自己的。我从未听说过这种方法成功的案例。

软件世界以外的人可能没有意识到，“Worse is Better”在整个艺术领域中都能找到。例如，在绘画中，这个想法是在文艺复兴时期被发现的。现在几乎每一位绘画老师都会告诉你，获得精确画作的正确方法不是慢慢地沿着物体的轮廓工作，因为错误会不断累积，最终你会发现线条对不上。相反，你应该先画几条大致在正确位置的快速线条，然后逐步完善这个初始草图。

在大多数领域中，原型传统上是用不同材料制作的。准备刻在金属上的字体最初是用画笔在纸上设计的。准备铸成青铜的雕像是在蜡上塑型的。准备绣在挂毯上的图案是用墨水在纸上绘制的。准备用石头建造的建筑则先用木材按较小比例进行测试。

当油画在十五世纪首次流行起来时，它如此令人兴奋的原因在于，你实际上可以直接从原型中制作出最终作品。如果你愿意，你可以先做一个初步的素描，但你不必受它束缚；你可以在完成绘画的过程中解决所有细节，甚至进行重大修改。

你在软件中也可以这样做。原型不必仅仅是一个模型；你可以把它逐步完善为最终产品。我认为只要可能，你就应该一直这样做。它让你能够利用一路上获得的新洞察。但或许更重要的是，这对士气有好处。

士气在设计中至关重要。我很惊讶人们没有更多地谈论它。我的第一位绘画老师告诉我：如果你在画某样东西时感到无聊，那么这幅画看起来就会很无聊。例如，假设你必须画一栋建筑，你决定逐块绘制每一块砖。如果你想这么做当然可以，但如果你画到一半感到无聊，开始机械地画砖而不是观察每一块，那么这幅画看起来会比你只是暗示砖块的存在更糟糕。

通过逐步完善原型来构建东西对士气有好处，因为它让你保持投入。在软件中，我的规则是：永远要有能运行的代码。如果你正在写的东西一个小时内就能测试，那么你就有立即获得回报的前景来激励自己。在艺术中也是如此，尤其是在油画中。大多数画家从模糊的草图开始，然后逐步完善它。如果你以这种方式工作，那么原则上你永远不必以看起来明显未完成的东西结束一天的工作。事实上，画家中甚至有这样一句话：“一幅画永远不会完成，你只是停止继续画它。”任何从事过软件工作的人都会对这个想法感到熟悉。

士气也是为什么很难为不老练的用户设计东西的另一个原因。很难对你自己不喜欢的东西保持兴趣。为了做出好东西，你必须想着“哇，这真的很棒”，而不是“这是什么垃圾；那些傻瓜会喜欢的。”

设计意味着为人类创造东西。但不仅仅用户是人类，设计师自己也是人类。

请注意，我一直以来谈论的都是“设计师”。设计通常必须由一个人掌控才能做出好的结果。然而，几个人的合作完成一个研究项目似乎是可能的。在我看来，这是研究与设计之间最有趣的区别之一。

艺术领域中存在著名的合作案例，但大多数似乎都是分子键合而非核聚变的例子。在歌剧中，通常由一个人写剧本，另一个人作曲。在文艺复兴时期，来自北欧的熟练工匠经常被雇来绘制意大利绘画背景中的风景。但这些都不是真正的合作。它们更像是罗伯特·弗罗斯特那句“好的篱笆造就好邻居”的例子。你可以把好的设计实例组合在一起，但在一个单独的项目内部，必须由一个人掌控。

我并不是说好的设计要求一个人想出所有东西。再没有比你信任其判断力的人给出的建议更有价值的东西了。但在讨论结束之后，关于要做什么的决定必须由一个人做出。

为什么研究可以由合作者完成而设计却不行？这是一个有趣的问题。我不知道答案。也许，如果设计和研究最终汇聚，那么最好的研究也是好的设计，而且事实上无法由合作者完成。许多最著名的科学家似乎都是独自工作的。但我了解得不够深入，无法判断这里是否存在某种模式。也可能只是因为许多著名科学家工作的时候，合作还没有那么普遍。

不管科学领域的故事如何，真正的合作在艺术中似乎极为罕见。由委员会设计是糟糕设计的同义词。为什么会这样？有没有办法打破这个限制？

我倾向于认为没有——好的设计需要一个独裁者。一个原因是好的设计必须是一个整体。设计不仅仅是为人类，而是为一个个具体的人。如果一个设计代表了一个能装进一个人头脑中的想法，那么这个想法也同样能装进用户的头脑中。
