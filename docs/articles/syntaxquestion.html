语法问题

阅读原文

（Guy Steele 和 Richard Gabriel 谈到了一个我预测会越来越多出现的问题，不过会来自相反的方向——当主流编程语言逐渐向 Lisp 靠拢之时。）

把类似 Algol 的语法引入 Lisp 的想法不断冒出来，而且几乎每次都会在两派人之间引发巨大争议：一派认为统一使用 S-表达式是一种技术优势（他们并不在意 S-表达式在数值表达式上相对笨拙这一公认的事实）；另一派则坚信代数语法更简洁、更方便，甚至更“自然”（不管这意味着什么，毕竟所有这些记号都是人为设计的）。

我们猜测，Algol 风格的语法之所以没有真正在整个 Lisp 社区流行开来，主要有两个原因。首先，可用的特殊符号不够用。当你的讨论领域仅限于数字或字符时，需要关注的运算种类有限，因此很容易为每个运算分配一个特殊字符并就此完事。但 Lisp 的领域要丰富得多，Lisp 程序员常常把一个应用当作一次新的语言设计练习；这种风格通常涉及设计新的数据结构以及操作它们的新函数——可能有几十甚至几百个——要发明那么多不同的符号实在太难了（尽管 APL 社区确实尝试过）。最终，人们总是不得不退回到通用的函数调用记法；区别只在于 Lisp 程序员不会等到失败之后才这么做。

其次，或许也更重要的是，Algol 风格的语法让程序看起来不再那么像用来表示它们的数据结构。在一个将程序表示的操控能力视为核心范式的文化中，一种让程序的外观与其作为数据时的表示形式相去甚远的记法，是不太可能受到热烈欢迎的（这过去是、现在依然是对 Common Lisp 中引入 loop 的主要反对理由之一）。

另一方面，正是因为 Lisp 让摆弄程序表示变得容易，初学者总是能轻松尝试各种替代记法。因此我们预计，未来几代的 Lisp 程序员会继续一遍又一遍地重新发明 Lisp 的 Algol 风格语法，而且我们同样确信，在最初的迷恋期过后，他们还会继续拒绝它。（或许这个过程应该被视为 Lisp 黑客的成人礼。）

- Steele 和 Gabriel，《Lisp 的演化》
