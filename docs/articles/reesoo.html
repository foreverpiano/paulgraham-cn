Rees 回复：OO

阅读原文

（Jonathan Rees 对 为什么 Arc 不是特别面向对象 给出了一个非常有趣的回应，他同意我在这里转载。）

下面是一份与这些术语相关的特性和属性的自助菜单；我听过人们把 OO 定义为这个列表中的很多不同子集。

封装（Encapsulation）——能够在语法层面隐藏类型的实现。例如在 C 或 Pascal 中，你总是能知道某个东西是 struct 还是数组，但在 CLU 和 Java 中，你可以把这种区别隐藏起来。

保护（Protection）——类型的客户端无法探测其实现。这保证了对实现进行行为保持的修改不会破坏其客户端，同时也确保了密码之类的敏感信息不会泄露。

特设多态（Ad hoc polymorphism）——函数和数据结构的参数可以接受多种不同类型的值。

参数多态（Parametric polymorphism）——函数和数据结构可以对任意值进行参数化（例如元素类型任意的列表）。ML 和 Lisp 都具备这一点。Java 则不完全具备，因为它存在非 Object 类型。

一切皆对象（Everything is an object）——所有值都是对象。这在 Smalltalk 中成立（？），但在 Java 中不成立（因为有 int 及其同伴）。

你唯一能做的就是发送消息（All you can do is send a message，简称 AYCDISAM）= Actor 模型——不存在对对象的直接操作，只有与对象的通信（或调用）。Java 中字段的存在违反了这一点。

规范继承 = 子类型化（Specification inheritance = subtyping）——语言中存在不同的类型，具有这样的性质：从类型正确性的角度看，一种类型的值可以当作另一种类型的值使用。（例如 Java 的接口继承。）

实现继承/复用（Implementation inheritance/reuse）——在写好一段代码之后，能以受控的方式生成一段相似的代码（例如它的超集），也就是说不必通过复制和修改来实现。这是一种有限且特殊的抽象形式。（例如 Java 的类继承。）

和式-积式-函数模式（Sum-of-product-of-function pattern）——对象（本质上）被限制为函数，该函数的第一个参数是一个特定的方法键，这个键来自一个有限的简单名称集合。

因此 OO 并不是一个定义明确的概念。有些人（例如 Abelson 和 Sussman？）说 Lisp 是 OO，他们指的是 {3,4,5,7}（前提是所有类型都存在于程序员的头脑中）。Java 被认为是 OO，是因为它具备 {1,2,3,7,8,9}。E 被认为比 Java 更 OO，因为它拥有 {1,2,3,4,5,7,9}，并且几乎具备 6；其中 8（子类化）被视为与 E 的目标相悖，而且对 OO 来说并非必要。

传统的类似 Simula 67 的类与实例模式能为你提供 {1,3,7,9}，我认为很多人把这当作 OO 的定义。

因为 OO 是一个不断移动的目标，OO 狂热者会随心所欲地挑选这个菜单中的某个子集，然后用它来试图说服你是个失败者。

造成困惑的部分原因——你在你的那篇小 为什么 Arc 不是特别面向对象 里也用另一种方式提到了——在于 C/C++ 阵营把 OO 看作是从一个完全没有第一类函数的世界中解放出来，而 Lisp 阵营则把 OO 视为牢笼，因为它把函数/对象的使用限制在了（9.）所描述的风格。在这种情况下，OO 唯一能被辩护的方式，就和任何其他游戏或戒律一样——通过论证你放弃了某些东西（例如向邻居家扔鸡蛋的自由），却获得了你真正想要的东西（确保邻居不会把你送进监狱）。

这也与 Lisp 面向孤独黑客，而施加纪律的语言则面向社会群体有关，这是你提到的另一点。在一个群体中，你希望尽可能限制其他人的自由，以减少他们干扰你或占你便宜的能力，而唯一能做到这一点的方法，要么是成为首领（危险且不太可能），要么是服从他们所遵守的相同规则。如果你服从规则，那么你会希望这些规则足够宽松，让你有机会做大多数自己想做的事，但又不能宽松到让别人能轻易攻击你。

在这样一个集体编程的世界里，语言就是一部宪法或一套规章制度，而解释器/编译器/QA 部门则部分扮演着规则检查者/执行者/警察的角色。共同编程者想要知道：如果我使用你的代码，它是会帮助我还是会伤害我？正确性是不可判定的（通常也无法强制执行），因此管理者只好采用项目启动时出现的任何一套规则（静态类型系统、语言限制、“lint”程序等等）。

我最近参与了 e-lang 邮件列表上关于反 OO 的讨论。我那条主要的反 OO 信息（实际上只攻击了第 5/6 点）是 http://www.eros-os.org/pipermail/e-lang/2001-October/005852.html。后续讨论很有意思，但我觉得它们没有完全正确地串成线程。

（以下是上面用到的几个术语的个人定义：

值（Value）= 可以传递给某个函数（抽象）的东西。（我排除了宏参数、参数化类型和模块之类的奇特编译期事物。）

对象（Object）= 具有函数式行为的值，即你可以对它调用方法、调用它、给它发消息之类的。有些人按照上面第 9 点的思路更严格地定义对象，而另一些人（例如 CLTL）则更宽松。这就是为什么在没有清晰定义的情况下，“一切皆对象”会变成一句空话。

在某些语言中，“调用”是柯里化的，键到方法的映射有时可以在编译期完成。这个技术细节会让关于 C++ 及相关语言中 OO 的讨论变得模糊。

函数（Function）= 可以与特定参数结合以产生某种结果的东西。在不同语言中，它可能与对象是同一回事，也可能不是。

类型（Type）= 对函数可以有意义地参数化的值空间的描述。我同时包括语言已知的类型，以及存在于程序员头脑中或文档里的类型。）
