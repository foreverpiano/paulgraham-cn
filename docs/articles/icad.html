} public static Inttoint foo(final int n) { return new Inttoint() { int s = n; public int call(int i) { s = s + i; return s; } }; } 它没有完全满足spec，因为它只能用于整数。在和Java黑客多次邮件往来之后，我认为要写出一个行为与前面例子一致的、真正支持多态的版本，难度介于“极其别扭”和“不可能”之间。如果有人愿意写一个，我会非常好奇地想看看，但我个人已经没精力继续了。 当然，说“在其他语言里无法解决这个问题”并不完全准确。所有这些语言都是图灵完备的，严格来说，你可以在其中任何一个语言里写出任何程序。那么你会怎么做呢？在极限情况下，通过在那个威力较弱的语言里写一个Lisp解释器。这听起来像个笑话，但它在大型编程项目中经常以不同程度发生，以至于有个专门的名字——Greenspun第十法则： 任何足够复杂的C或Fortran程序，都包含一个临时拼凑的、不正式指定的、充满bug的、运行缓慢的Common Lisp的半成品实现。 如果你想解决一个困难的问题，问题不在于你是否会使用足够强大的语言，而在于你是否会：(a) 使用一门强大的语言，(b) 为一门强大的语言写一个事实上的解释器，或者(c) 自己变成一门强大语言的人肉编译器。 我们在上面的Python例子中已经看到这种情况开始发生——我们实际上是在模拟编译器为实现词法变量而会生成的代码。这种做法不仅常见，而且已经制度化了。例如，在面向对象的世界里，你会经常听到“模式”（patterns）这个词。我怀疑这些模式有时正是(c)情况——人肉编译器——在工作的证据。当我在自己的程序里看到模式时，我会把它视为麻烦的信号。程序的形状应该只反映它需要解决的问题。代码中任何其他的规律性，在我看来至少说明我正在使用的抽象还不够强大——常常意味着我正在手动展开某个我本该写成宏的东西。 注 IBM 704的CPU大约有一台冰箱那么大，但要重得多。CPU重3150磅，而4K的内存则装在另一个重达4000磅的箱子里。Sub-Zero 690是目前最大的家用冰箱之一，重656磅。 Steve Russell还在1962年写了第一个（数字）计算机游戏《Spacewar》。 如果你想骗一个“尖头老板”让你用Lisp写软件，可以试着告诉他这是XML。 下面是用其他Lisp方言写的累加器生成器： Scheme: (define (foo n) (lambda (i) (set! n (+ n i)) n)) Goo: (df foo (n) (op incf n _))) Arc: (def foo (n) [++ n _]) Erann Gat在JPL关于“行业最佳实践”的悲惨故事，促使我对这个经常被误用的说法进行了一般性讨论。 Peter Norvig发现《设计模式》一书中的23个模式，有16个在Lisp中“看不见或更简单”。 感谢许多回答我关于各种语言的问题和/或阅读过本文草稿的人，包括Ken Anderson、Trevor Blackwell、Erann Gat、Dan Giffin、Sarah Harlin、Jeremy Hylton、Robert Morris、Peter Norvig、Guy Steele和Anton van Straaten。他们对文中表达的任何观点不承担责任。 相关阅读：很多人对这次演讲做出了回应，因此我专门建立了一个额外页面来处理他们提出的问题：Re: Revenge of the Nerds。它还在LL1邮件列表上引发了广泛且常常很有价值的讨论。特别推荐Anton van Straaten关于语义压缩的那封邮件。 LL1上的某些讨论促使我进一步深入思考语言威力的问题，写成了《简洁即是力量》。 一套更完整的累加器生成器基准测试的规范实现被收集在单独的一页上。 日文翻译、西班牙文翻译、中文翻译 你可以在《黑客与画家》一书中找到这篇文章以及另外14篇散文。
