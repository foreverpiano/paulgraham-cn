Arc 发布了

January 2008 · 阅读原文

2008 年 1 月 29 日

我们今天发布了 Arc 的一个版本，同时在 arclanguage.org 上建立了介绍它的网站。这个网站对 Hacker News 的用户来说会非常熟悉。它基本上使用相同的代码，只是改了一些颜色和文案。

Arc 仍然是一个正在进行中的项目。我们只是把当前的代码拍了个快照放到网上而已。我花了几天时间清理了一些不一致的地方，但它仍处于大多数软件都会有的半成品状态，里面充满了各种 hack 和“以后要修”的备注。

为什么要现在发布？因为几个月前我突然意识到，它已经足够好了。即使还远未完成，我在编写大多数程序时也更愿意用 Arc 而不是 Scheme 或 Common Lisp。而我作为一名有着多年使用经验的 Lisp 程序员，算是相当有代表性。所以虽然 Arc 不是完美的 Lisp，但至少在某些类型的编程上，它似乎比目前两种主流选择都要好。

我对发布它有些担心，因为我不希望有任何力量迫使这门语言停止演变。一旦你发布了一样东西，人们开始在上面构建东西，你就会开始觉得不该再改动了。所以我们提前声明：我们会继续像只有自己在用它一样去开发。我们会毫不犹豫地修改东西，甚至不会去记录改动了什么。

我知道这听起来很刺耳，但这里面的利害关系很大。去年夏天我去听 Guido van Rossum 关于 Python 的演讲，他似乎把前一年大部分时间都花在了从一种字符表示切换到另一种上。我绝不想花一整年去处理字符问题。为什么 Guido 不得不这么做？因为他必须考虑兼容性。虽然担心破坏现有代码听起来很负责任，但最终是要付出代价的：你花一年时间去对付字符集，而不是让语言变得更强大。

这也是为什么 Arc 目前只支持 Ascii 的原因。当前版本的 Arc 编译到 MzScheme 上，后者对字符处理有更先进的方案。但要弄清楚怎么和它对接可能要花我几天时间，而我连一天都不想花在字符集上。字符集是个黑洞。我知道只支持 Ascii 很不国际化，甚至到了几乎冒犯人的地步，就像把 Beijing 叫成 Peking，或者把 Roma 叫成 Rome 一样（嗯，等一下）。但会被这种事冒犯的人，本来也不会喜欢 Arc。

Arc 对 HTML 也抱有类似的不政治正确的态度。预置的库统一用表格来实现所有布局。为什么？因为 Arc 是为探索式编程量身定制的，而 W3C 认可的做法代表了完全相反的精神。

用列表来表示事物和用带有命名、类型字段的“对象”来表示事物之间也存在类似的对立。在我用 Lisp 编程两三年后，我曾经经历过一个阶段，认为用列表表示一切的古老方式只是个 hack。如果你需要表示点，显然声明一个带有 x 和 y 字段的正式结构，要比用一个包含两个数字的列表更好。列表可以包含任何东西，甚至元素数量都可能不一样。

我错了。这些恰恰就是用列表表示点的优势。

这些年来，我对列表的欣赏与日俱增。在探索式编程中，一个列表到底代表什么并不清晰，这反而是一种优势，因为你自己对正在尝试编写哪种类型的程序也并不清楚。最重要的是不要限制自己想法的演化。因此，你在代码中对数据结构的含义承诺得越少越好。

表格就是 HTML 中的列表。W3C 不喜欢你用表格做超出显示表格数据的事，因为那样一个表格单元格的含义就不清晰了。但这种模糊性并不总是错误。它可能准确反映了程序员当下的思维状态。在探索式编程中，程序员按定义就是不确定程序到底要表达什么的。

当然，“探索式编程”其实就是“快速而脏”的编程的委婉说法。而这个短语几乎是多余的：快速几乎总是意味着脏。写快速而脏的程序总是让人有点不好意思。然而，一些（如果不是大多数）最好的程序都是这样开始的。而一些（如果不是大多数）最惊人的软件失败，都是由试图反其道而行之的人造成的。

所以经验告诉我们，应该拥抱“脏”。或者至少是某些形式的脏；在其他方面，最好的快速而脏的程序通常都相当干净。哪种脏是坏的，哪种是好的？最好的快速而脏的程序，似乎是那些数学上优雅，但缺少一些功能——尤其是那些非本质却被认为出于礼节必须具备的功能的程序。好的干净是对问题本身施加的约束的回应。坏的干净则是对来自外部的约束的回应——来自规章制度，或强大组织的期望。

我认为这两种干净不仅不同，而且是相互对立的。“规则”无论是什么，通常都是由政治决定的；你只有牺牲数学上的优雅才能遵守它们。反之亦然。

Arc 试图成为一门在正确的地方脏的语言。例如，它尽量不去禁止事情。凡是我发现自己在问“应该允许人们……吗？”的地方，我都尽量回答“是”。这不是那种试图保护程序员免于自伤的语言。

Arc 试图避免的那种脏，是冗长、重复的源代码。避免它的方法不是禁止程序员这么写，而是让写出简洁代码变得容易。在编写 Arc 的过程中，我做的一件事就是反复审视各种应用，问自己：我能对语言做些什么来让这段代码更短？当然不是指字符数或行数，而是指 token 数。从某种意义上说，Arc 是多年积累的让程序更短的技巧的集合。听起来没什么雄心壮志，但这其实正是高级语言的目的：让程序更短。

在正确的地方脏，意味着放纵却又优雅。我不知道 Arc 现在是否已经能诚实地用这样诱人的词语来形容，但这就是我们的目标。目前最好说，它是一门用来写快速而脏的程序的、快速而脏的语言。
