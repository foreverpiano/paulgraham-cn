更好的贝叶斯过滤

January 2003 · 阅读原文

2003年1月

（本文是2003年垃圾邮件大会上的演讲内容。它描述了我为改进 反垃圾邮件计划 一文中提出的算法性能所做的工作，以及我未来计划要做的事。）

我想在这里介绍的第一个发现是一种懒惰评估研究论文的算法。你只需随便写点什么，不引用任何前人工作，那些愤愤不平的读者就会主动把你本该引用的所有论文参考资料发给你。我是在“[1]A Plan for Spam”被 Slashdot 报道后发现这个算法的。

垃圾邮件过滤是文本分类的一个子领域，这是一个早已确立的领域，但专门讨论贝叶斯垃圾邮件过滤的第一批论文似乎是1998年在同一会议上发表的两篇，一篇来自 Pantel 和 Lin [2]，另一篇来自微软研究院的一个小组 [3]。

当我听说这项工作时，感到有些惊讶。如果人们早在四年前就已开始研究贝叶斯过滤，为什么没有普及开来呢？读完那些论文后我明白了原因。Pantel 和 Lin 的过滤器是两者中效果更好的，但它只拦截了92%的垃圾邮件，误判率为1.16%。

而当我尝试编写一个贝叶斯垃圾邮件过滤器时，它拦截了99.5%的垃圾邮件，误判率低于0.03% [4]。当两个人做同一个实验却得到截然不同的结果时，总会让人警觉。这里尤其令人警觉，因为这两组数字可能会得出完全相反的结论。不同用户有不同的要求，但我认为对很多人来说，92%的过滤率和1.16%的误判率意味着过滤不是一个可接受的方案，而99.5%的过滤率和低于0.03%的误判率则意味着它是可行的。

那么为什么我们会得到如此不同的数字呢？我没有尝试重现 Pantel 和 Lin 的实验，但从论文中我看到了五个可能导致差异的原因。

首先，他们用来训练过滤器的数据非常少：只有160封垃圾邮件和466封正常邮件。在如此小的数据集上，过滤器的性能应该仍在持续提升。因此他们的数字甚至可能无法准确衡量其算法的性能，更不用说代表贝叶斯垃圾邮件过滤的整体水平了。

但我认为最重要的差异可能是他们忽略了邮件头信息。对任何做过垃圾邮件过滤器的人来说，这似乎是一个反常的决定。然而在我最早尝试编写的过滤器中，我也忽略了邮件头。为什么？因为我想让问题保持简洁。那时我对邮件头了解不多，在我看来它们充满了随机内容。这里给过滤器开发者一个教训：不要忽略数据。你可能会觉得这个教训太过明显无需提及，但我自己却不得不多次吸取这个教训。

第三，Pantel 和 Lin 对 token 进行了词干提取，也就是把“mailing”和“mailed”都简化为词根“mail”。他们可能是因为语料库太小而不得不这么做，但如果是这样，这其实是一种过早的优化。

第四，他们计算概率的方法不同。他们使用了所有 token，而我只使用15个最重要的 token。如果你使用所有 token，就容易漏掉较长的垃圾邮件——那种讲完人生故事直到告诉你他通过多级营销致富的邮件。而且这样的算法很容易被垃圾邮件发送者欺骗：只要添加一大段随机文本来平衡垃圾词的影响即可。

最后，他们没有对误判进行偏置处理。我认为任何垃圾邮件过滤算法都应该有一个方便的旋钮，可以通过牺牲过滤率来降低误判率。我的做法是将正常邮件语料库中出现的 token 计数翻倍。

我不认为把垃圾邮件过滤当作一个单纯的文本分类问题是个好主意。你可以使用文本分类技术，但解决方案应当反映出这些文本是电子邮件这一事实，尤其是垃圾邮件这一事实。电子邮件不仅仅是文本，它具有结构。垃圾邮件过滤也不仅仅是分类，因为误判的后果远比漏判严重得多，因此应当将其视为一种不同类型的错误。而且错误的来源不只是随机变化，而是一个活生生的人类垃圾邮件发送者在积极地试图击败你的过滤器。

Token

在 Slashdot 文章之后，我听说的另一个项目是 Bill Yerazunis 的 CRM114 [5]。这是对我刚才提到的设计原则的反例。它是一个纯粹的文本分类器，但效果惊人，以至于它几乎完美地过滤了垃圾邮件，却甚至不知道自己在做什么。

一旦我理解了 CRM114 的工作原理，我就知道我最终必须从基于单个单词的过滤转向类似的方法。但我想先看看使用单个单词能走多远。结果是，令人惊讶的是，能走得很远。

我主要在研究更智能的 token 切分方法。在当前的垃圾邮件上，我已经能达到接近 CRM114 的过滤率。这些技术与 Bill 的方法大多是正交的；一个最优的解决方案可能会同时采用两者。

“A Plan for Spam”中使用了一个非常简单的 token 定义。字母、数字、连字符、撇号和美元符号是组成部分，其他所有字符都是 token 分隔符。我还忽略了大小写。

现在我有了一个更复杂的 token 定义：

保留大小写。

感叹号是组成部分。

句点和逗号如果出现在两个数字之间则视为组成部分。这让我能完整保留 IP 地址和价格。

像 $20-25 这样的价格范围会产生两个 token：$20 和 $25。

出现在 To、From、Subject 和 Return-Path 行中，或出现在 URL 中的 token，会相应地被标记。例如，Subject 行中的“foo”会变成“Subject*foo”。（星号可以是你不允许作为组成部分的任意字符。）

这些措施增加了过滤器的词汇量，使其区分能力更强。例如，在当前的过滤器中，Subject 行中的“free”垃圾概率为98%，而在正文中的同一个 token 垃圾概率只有65%。

以下是一些当前的概率 [6]：

Subject*FREE 0.9999 free!! 0.9999 To*free 0.9998 Subject*free 0.9782 free! 0.9199 Free 0.9198 Url*free 0.9091 FREE 0.8747 From*free 0.7636 free 0.6546

在 Plan for Spam 的过滤器中，所有这些 token 的概率都会相同，都是0.7602。那个过滤器识别出大约23,000个 token。目前的过滤器能识别大约187,000个。

拥有更大 token 集合的缺点是漏匹配的机会增加。将你的语料分散到更多 token 上，其效果与缩小语料库一样。例如，如果你将感叹号视为组成部分，那么你最终可能没有“free”后面跟着七个感叹号的垃圾概率，尽管你知道只有两个感叹号的“free”概率是99.99%。

解决这个问题的一个方法是我称之为“退化”的技术。如果你找不到某个 token 的精确匹配，就把它当作一个不那么具体的版本来处理。我认为末尾的感叹号、大写字母，以及出现在五个标记上下文之一中，都会使一个 token 变得更具体。例如，如果我找不到“Subject*free!”的概率，我会查找“Subject*free”、“free!”和“free”的概率，并取其中离0.5最远的那个。

以下是当过滤器在 Subject 行中看到“FREE!!!”且没有其概率时会考虑的备选方案 [7]。

Subject*Free!!! Subject*free!!! Subject*FREE! Subject*Free! Subject*free! Subject*FREE Subject*Free Subject*free FREE!!! Free!!! free!!! FREE! Free! free! FREE Free free

如果你要这么做，一定要同时考虑首字母大写、全大写和全小写的版本。垃圾邮件往往会有更多祈使语气的句子，而这些句子的第一个词是动词。因此首字母大写的动词比全小写时具有更高的垃圾概率。在我的过滤器中，“Act”的垃圾概率是98%，而“act”只有62%。

如果你增加过滤器的词汇量，按照你旧的“相同”定义，可能会多次统计同一个词。从逻辑上讲，它们已经不是同一个 token 了。但如果这仍然让你不安，我可以根据经验告诉你，那些你似乎多次统计的词，往往正是你想要多次统计的那些。

更大词汇量的另一个效果是，当你查看 incoming 邮件时，会发现更多有趣的 token，也就是那些概率远离0.5的 token。我使用15个最有趣的 token 来判断邮件是否为垃圾邮件。但当你使用这样一个固定数量时，可能会遇到问题。如果你发现大量最大程度有趣的 token，最终结果可能会由决定同等有趣 token 排序的随机因素来决定。解决这个问题的一个方法是将某些 token 视为比其他更感兴趣。

例如，token“dalco”在我的垃圾邮件语料中出现了3次，在合法邮件中从未出现。而 token“Url*optmails”（表示 URL 中的“optmails”）出现了1223次。然而，按照我过去计算 token 概率的方法，这两个 token 会有相同的垃圾概率，即0.99的阈值。

这感觉不对。从理论上讲，给这两个 token 赋予显著不同的概率是有依据的（Pantel 和 Lin 就是这么做的），但我还没有尝试过。至少如果我们发现超过15个只出现在一个语料中的 token，我们应该优先考虑那些出现次数多的。所以现在有了两个阈值。对于只出现在垃圾邮件语料中的 token，如果出现次数超过10次，概率为0.9999，否则为0.9998。在合法邮件语料端也是如此。

我以后可能会大幅缩放 token 概率，但这一点微小的缩放至少确保 token 能按正确的方式排序。

另一种可能是不仅考虑15个 token，而是考虑所有超过某个有趣度阈值的 token。Steven Hauser 在他的统计垃圾邮件过滤器中就是这么做的 [8]。如果你使用阈值，就要把阈值设得很高，否则垃圾邮件发送者可以通过在消息中塞满更多无害词语来欺骗你。

最后，对于 HTML 该怎么处理呢？我尝试了从完全忽略到完全解析的各种方案。忽略 HTML 是个坏主意，因为它充满了有用的垃圾邮件信号。但如果你完全解析它，你的过滤器可能会退化成一个单纯的 HTML 识别器。最有效的方法似乎是中间路线：注意某些 token，而忽略其他。我会查看 a、img 和 font 标签，忽略其余的。链接和图片当然要查看，因为它们包含 URL。

我也许能在处理 HTML 上做得更聪明，但我认为不值得投入大量时间。充满 HTML 的垃圾邮件很容易过滤。更聪明的垃圾邮件发送者已经避免使用它。因此未来的性能应该不会过多依赖于你如何处理 HTML。

性能

在2002年12月10日至2003年1月10日期间，我收到了大约1750封垃圾邮件。其中4封漏过了。过滤率约为99.75%。

四封漏过的垃圾邮件中有两封是因为它们恰好使用了在我合法邮件中经常出现的词。

第三封是那种利用不安全的 CGI 脚本向第三方发送邮件的类型。仅基于内容很难过滤它们，因为它们的邮件头看起来很正常，而且用词也很谨慎。尽管如此我通常还是能抓住它们。这一封以0.88的概率勉强通过，刚好低于0.9的阈值。

当然，如果查看多个 token 序列就能轻易抓住它。“Below is the result of your feedback form”是一个立刻暴露身份的短语。

第四封垃圾邮件是我称之为“未来垃圾邮件”的类型，因为我预计垃圾邮件会演化成这种形式：一些完全中性的文本后面跟着一个 URL。在这个例子中，邮件来自某人说他终于完成了自己的主页，问我是否愿意去看一看。（那个页面当然是一个色情网站的广告。）

如果垃圾邮件发送者小心处理邮件头并使用全新的 URL，那么“未来垃圾邮件”中就没有过滤器能注意到的内容。当然，我们可以通过派出一个爬虫去查看那个页面来反制。但这可能不是必要的。“未来垃圾邮件”的响应率一定很低，否则大家都会这么做。如果低到一定程度，垃圾邮件发送者发送它就不划算了，我们也不需要太费力去过滤它。过滤器会消灭垃圾邮件吗？

现在来说真正令人震惊的消息：在同一个月的时间里，我只出现了三次误判。

从某种意义上说，出现一些误判反而让人松了一口气。当我写“A Plan for Spam”时还没有遇到过误判，也不知道它们会是什么样子。现在我遇到了一些，发现它们并不像我担心的那么糟糕。统计过滤器产生的误判结果表明，它们是那些听起来很像垃圾邮件的邮件，而这些往往正是你最不介意漏掉的那些 [9]。

两次误判是来自我曾经买过东西的公司的新闻通讯。我从未要求接收它们，所以从某种角度说它们算是垃圾邮件，但我把它们算作误判，因为我之前并没有把它们当作垃圾邮件删除。过滤器抓住它们的原因是，这两家公司在一月份都改用商业邮件发送服务，而不是从自己的服务器发送邮件，于是它们的邮件头和正文都变得更像垃圾邮件了。

不过第三次误判比较严重。它来自埃及某人，全篇都是大写字母。这是让 token 对大小写敏感的直接结果；Plan for Spam 的过滤器不会抓住它。

很难说整体误判率是多少，因为从统计上讲我们已经处于噪声范围内。任何做过过滤器的人（至少是有效的过滤器）都会意识到这个问题。有些邮件很难说它们是不是垃圾邮件，而当你把过滤器调得非常紧时，看到的正是这些邮件。例如，到目前为止过滤器抓住了两封因为打字错误发到我地址的邮件，以及一封以为我是别人而发给我的邮件。从某种角度说，这些既不是我的垃圾邮件，也不是我的正常邮件。

另一次误判来自 Virtumundo 的一位副总裁。我假装客户给他们写信，而回复是通过 Virtumundo 的邮件服务器回来的，因此带有最能定罪的邮件头。从某种角度说这也不是一个真正的误判，而是一种类似海森堡不确定性效应的东西：我之所以收到它，是因为我正在写关于垃圾邮件过滤的文章。

不算这些，我总共出现了五次误判，来自大约7740封合法邮件，误判率为0.06%。另外两次是一封我购买的商品缺货的通知，以及一封来自 Evite 的聚会提醒。

我认为这个数字不可靠，一部分是因为样本太小，另一部分是因为我觉得我可以修复过滤器，让它不再抓住其中一些。

在我看来，误判和漏判是不同类型的错误。过滤率是对性能的衡量。而我认为误判更像是 bug。我把提高过滤率视为优化，而把减少误判视为调试。

所以这五个误判就是我的 bug 列表。例如，那封来自埃及的邮件被抓住，是因为大写文本让过滤器觉得它像一封尼日利亚垃圾邮件。这确实算是一个 bug。就像 HTML 一样，整封邮件都是大写实际上在概念上是一个特征，而不是每个词一个。我需要用更复杂的方式处理大小写。

那么该如何看待这个0.06%呢？我认为没多少意义。你可以把它当作一个上限，但要记住样本量很小。但在这个阶段，它更多地衡量的是我实现中的 bug，而不是贝叶斯过滤的内在误判率。

未来

接下来做什么？过滤是一个优化问题，而优化的关键是性能分析。不要试图猜测你的代码哪里慢，因为你会猜错。看看你的代码哪里慢，然后修复那里。在过滤中，这意味着：看看你漏掉的那些垃圾邮件，想想你本来可以做什么来抓住它们。

例如，垃圾邮件发送者现在正积极地尝试规避过滤器，他们正在做的一件事是拆分和拼错单词以防止过滤器识别它们。但这不是我的首要任务，因为我仍然能毫不费力地抓住这些垃圾邮件 [10]。

目前我确实有麻烦处理的两类垃圾邮件。一类是假装来自女性，邀请你和她聊天或查看她在约会网站的资料的那种。这些能通过是因为它们是你可以在不使用销售话语的情况下进行的唯一一种推销方式。它们使用的词汇和普通邮件一样。

我过滤有困难的另一类垃圾邮件是来自例如保加利亚的公司提供的合同编程服务。这些能通过是因为我自己也是程序员，而这些垃圾邮件充满了和我真实邮件相同的词。

我可能会先重点关注个人广告类型。我认为如果我仔细观察，应该能找到这些邮件与我的真实邮件之间的统计差异。写作风格肯定是不同的，虽然可能需要多词过滤才能捕捉到这一点。另外，我注意到它们倾向于重复 URL，而正常邮件中包含 URL 的人通常不会这么做 [11]。

外包类型的会很难抓住。即使你派一个爬虫去那个网站，你也不会找到确凿的统计证据。也许唯一的答案是建立一个在垃圾邮件中被宣传的域名中央列表 [12]。但这种类型的邮件不可能有很多。如果剩下的唯一垃圾邮件是来自保加利亚的主动提供的合同编程服务，我们大家大概就可以去研究其他东西了。

统计过滤真的能把我们带到那个地步吗？我不知道。目前对我个人来说，垃圾邮件不是一个问题。但垃圾邮件发送者还没有认真尝试欺骗统计过滤器。当他们这么做时会发生什么？

我对在网络层工作的过滤器并不乐观 [13]。当存在一个值得绕过的静态障碍时，垃圾邮件发送者会非常高效地绕过它。已经有一家叫 Assurance Systems 的公司会把你的邮件通过 Spamassassin 运行一遍，然后告诉你它是否会被过滤掉。

网络层过滤器不会完全没用。它们可能足以消灭所有“opt-in”垃圾邮件，也就是来自 Virtumundo 和 Equalamail 之类声称自己在运行 opt-in 列表的公司的垃圾邮件。你可以仅基于邮件头来过滤它们，不管它们在正文里说什么。但任何愿意伪造邮件头或使用开放中继的人（大概包括大多数色情垃圾邮件发送者），如果他们想的话，应该能够让某些消息绕过网络层过滤器。（尽管可能不是他们想发送的那种消息，这也算聊胜于无。）

我比较乐观的那类过滤器是基于每个用户个人邮件计算概率的过滤器。这些可以有效得多，不仅能避免误判，在过滤上也能更强：例如，在消息中任何位置发现收件人 email 地址的 base-64 编码就是一个非常好的垃圾邮件指标。

但个人过滤器的真正优势在于它们都会不同。如果每个人的过滤器都有不同的概率，这会让垃圾邮件发送者的优化循环——程序员称之为编辑-编译-测试周期——变得极其缓慢。他们不能仅仅调整一封垃圾邮件直到它能通过他们桌面上的某个过滤器副本，而是必须为每次调整都做一次测试群发。这就像在没有交互式顶层环境的语言中编程，我不会希望任何人经历这种事。
