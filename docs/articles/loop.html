Dan Weinreb：Loop 宏

阅读原文

（在回复 LL1 邮件列表中对 Waters 系列宏的引用时，Common Lisp 的设计者之一 Dan Weinreb 谈到了 Loop 宏。我认为 Loop 是 CL 中最糟糕的缺陷之一，无论是宏的编写者还是语言设计者，都应该将其作为前车之鉴。）

这大概是在 1978-1979 年左右，当时我们都想对 Maclisp 的“新式 DO”进行泛化和扩展，这是 LOOP 的主要竞争对手。每种方案都有自己的优点和缺点。我认为 LOOP 阵营的我们可能对 Waters 方法的优点不够认同，一方面是因为我们当时对词法作用域还不够熟悉，另一方面也是因为他的方案确实在某些方面不如 LOOP 做得好。

顺便说一句，LOOP 是我们故意违反 Scott McKay 最近在这个列表上列举的一条（极好的！）规则的少数宏之一。具体来说，我们（“我们是谁啊？”，其实主要是 Dave Moon，但我们其他人也在怂恿他）决定打破 Lisp 特殊形式通常必须以列表的第一个元素为符号、后续“语法”完全通过嵌套列表结构来表达的严格限制。

毕竟，我们是引入 Lisp 关键字参数的人。那些是在运行时以某种语法方式使用的符号。这里我们更进一步，在“编译时”——即静态地、语法性地（例如不需要用 QUOTE）——使用符号作为关键字。于是我们创造了 Lisp 中最早的以关键字为导向的特殊形式之一，我认为它可能是 Common Lisp 中唯一被接受的这类特殊形式。

（而它也是在经历了非常惨烈的争斗之后才被接受的，这一点 Guy Steele 比任何人都清楚，因为他是那些争斗的裁判！）

LOOP 并没有像我们希望的那样成功。直到今天，我仍然坚定支持我们使用关键字的决定。所有规则都有例外，这是一个使用关键字的好处超过采用不寻常语法代价的例子。使用关键字并不是 LOOP 的问题所在。

相反，LOOP 的问题在于，当你开始把 LOOP 的许多不同特性组合在一起使用时，很难预测它会做出什么行为。这是个严重的问题，因为 LOOP 的整个设计理念就是让你可以把多种特性一起使用；如果不这么做，LOOP 就显得杀鸡用牛刀了。Moon 曾非常努力地想让 LOOP 变得更可预测、更易理解。即使在当时，我也并没有完全理解所有问题；现在，二十多年过去了，我几乎已经忘记了。但如果连 Dave Moon 都无法找出解决办法，我非常确信这是一个极难解决的问题。
