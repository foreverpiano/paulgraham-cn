Arc 三周

November 2001 · 阅读原文

2001 年 11 月

1. 说明：

- Arc 尚未完成。

- 欢迎提出建议。

- 别太震惊（你可能会慢慢习惯）。

Arc 是 LL1 会议上展示的最年轻的语言。它只有大约三周的历史。大多数语言在这个阶段可能看起来都很糟糕，但我希望尽早向大家展示 Arc，以听取他们的意见。

Arc 中的一些想法可能会让人震惊，尤其是对编译器作者来说。所以请先忍耐一下。其中一些想法随着时间推移会变得不那么惊人。另一些可能确实不行，我们会重新做。

2. Lisp

- 一门拥有方言的语言。

- 80 年代中期以来没有出现过新的 Lisp（当时也算不上新）。

- 如今语言环境已大不相同：Unix 胜出、庞大的库、活跃的开发。

- 清漆里不放洋葱。

Lisp 是一种不寻常的语言，因为它拥有多种方言。Lisp 依赖于一组小而明确的核算符，任何包含这些算符的语言都是 Lisp 的一个方言——这不仅仅是社会习俗，而是在形式意义上：如果你拥有这些核算符，就可以定义出其余的一切。如果你设计了一门语言，它拥有 car、cdr、cons、quote、eq、cond，以及用 cons 构造函数的记法，那么你就设计出了 Lisp 的一个方言，即使你并非有意如此。

是时候推出一个新的 Lisp 方言了。目前领先的两个方言 Common Lisp 和 Scheme，自 20 世纪 80 年代以来都没有发生实质性的改变。而自那时起，编程语言的含义已经发生了变化。1985 年，一门编程语言只是一份规格说明。如今，得益于 Perl，它不仅（甚至可能主要不是）指一份规格，还包括一个优秀的免费实现、庞大的库以及持续的更新。

自 1985 年以来另一点改变是：Unix 赢了。因此现在大家对操作系统能提供什么有了更多共识。Common Lisp 和 Scheme 诞生于语言必须保持操作系统中立的时代。一门编程语言不能有 socket 的概念，例如：要是底层操作系统没有 socket 怎么办？现在情况变了。一门不允许你打开 socket 的语言显得几乎是刻意刁难。

如果你只是做一门能与操作系统对话、并拥有强大字符串库的 Lisp，我认为它已经足以立足。不过我们希望做得更多。Arc 的目标不是更新 Common Lisp 或 Scheme。在 Arc 中，我们打算回到 Lisp 的源头——McCarthy 1960 年的论文，从最底层重新构建这门语言。

Primo Levi 在《元素周期表》中讲了一个他在清漆工厂工作时的故事。他是化学家，对清漆配方里居然加了一颗生洋葱这件事很着迷。它是用来干什么的？没人知道，它只是配方的一部分。于是他去研究，最终发现他们多年前开始扔洋葱是为了测试清漆的温度：如果够热，洋葱就会被炸熟。

我们打算尽量不在 Arc 里放任何“洋葱”。一切都可质疑。例如，在 Arc 中，lambda 被叫做 fn。一开始这个想法让我震惊，但我觉得 fn 更短，至少同样富有表达力。要是我只是习惯了 lambda 呢？于是，带着一种不安的责任感，

我决定试试看。几天之后我居然更喜欢 fn 了。现在在我看来，lambda 显然是一颗洋葱：如果 Alonzo Church 每次都要把 lambda 这个词写出来，他自己也不会用的。

3. 为优秀程序员设计的语言

- 目标用户：与 Java 相反。

- 可编程的编程语言。

- 默认允许。

- 我希望有人为我打造的语言。

- 简洁（你喜欢抽象的原因）

正如 Gosling 在第一份 Java 白皮书中所说，Java 是为普通程序员设计的。为普通程序员设计语言是一个完全合理的目标。（或者说，为小孩子设计语言，比如 Logo。）但为设计理念设计语言也是一个合理且非常不同的目标。

为普通程序员设计的语言必须把安全性放在首位。而专家级程序员只关心能力，会对任何以安全为名阻碍他们的语言感到恼火。你可以在任何工具上看到这种差异，从汽车到洗碗机都是如此。

我不知道以前是否有人有意识地为一群优秀的程序员设计语言。但确实有几门语言实际上是为优秀程序员设计的，因为优秀程序员是为自己使用而设计它们的。C、Lisp 和 Smalltalk 都是这样诞生的。

当你可以假设用户是优秀程序员时，问题在某些方面会变得更容易。语言设计者常常担心如果允许用户做某某事，他们可能会搞出一团糟。一旦假设用户是优秀程序员，你就自动有了所有这类问题的答案：让用户做他想做的任何事。

Lisp 一直处于这个连续谱的极端一端。Lisp 是一只变色龙称它为“可编程的编程语言”，因为程序员可以改变的东西实在太多了。Arc 的目标就是最大化 Lisp 的这一特性。我们不再假设自己知道什么东西允许程序员接触是好的，而是假设程序员会想做我们甚至从未想象过的事情，因此需要能够接触到一切。

优秀程序员还喜欢简洁，这是 Arc 的另一个主要目标。你经常听到说编程语言好是因为它们提供了抽象。我认为我们真正喜欢的不是抽象本身，而是简洁。如果一种表达程序的方式更加抽象，却让你的程序更长，那就没什么吸引力了。（这不是一个假设的例子，它在 Prolog 中就发生了。）

我所说的简洁不是指程序应该包含更少的字符。那固然有一定价值，但更重要的是需要更少的词法记号（token）。

Perl 是简洁的一个鼓舞人心的例子。Larry Wall 打破了所有规则，并在此过程中发现了一些好主意。Perl 可能是个拼凑的怪物，但它能让你的程序变短，你必须尊重这一点。

在 Arc 中，我们希望让程序一样短甚至更短，同时以透明的方式从清晰理解的基础之上构建语言。我们这样做不仅仅是出于挑剔。如果你想让用户能够定制它，就必须透明地构建语言。Perl 那种混乱的语义会让加入宏变得非常困难。

4. 其他原则

- 去做程序员实际上（偷偷）想要的事。

- 代码的外观很重要：短名字，不要用咒骂符号

- 多态：(+ "foo" "bar") -> "foobar"

- 特别适合 Web 应用。

- Perl 的教训：代词。

我们为 Arc 还有另外几个设计目标。我们时刻记住语言是为程序员服务的，因此应该做程序员想要的事。程序员实际想要的，可能和语言设计者认为的好设计不是一回事。在这种情况下，语言设计者应该抛开自己的原则，去倾听程序员的想法。

你不会想成为那种现代主义建筑师，设计一把全是直角的椅子，只为了符合某种预设的好设计理念。那样你只是在解决错误的问题。椅子（除了少数明确设计为雕塑的）是给人坐的。

我曾问一位朋友——他几乎是你能遇到的最好的程序员——关于创建新的局部变量的事。他有些犹豫地承认，他不喜欢传统的 Lisp let，因为它同时引入了一层新的缩进。他之所以犹豫，是因为他知道在原则上 let 是“正确”的东西。这时你必须问：“对谁正确？”因为我认为没有哪位程序员明显比这个人更聪明。如果他想以某种方式做事，那就是你能得到的最好的“好设计”检验——肯定比任何抽象原则都更好。

程序员不愿意承认但几乎都强烈感觉到的一点是：代码的外观很重要。是的，它确实很重要。我们不仅要努力让 Arc 变得漂亮，还要让你能在你对美的看法不同时，改变这门语言的外观。

软件设计者通常要么属于短名称派，要么属于长名称派。Unix 和 C 偏好短名称。Common Lisp 和 Smalltalk 则属于对立阵营。长名称的理由是它们更有描述性，因此让程序员（尤其是初学者）更容易记住底层算符的作用。反对长名称的理由是它们会让程序变得杂乱。这里我们再次被“用户是优秀程序员”这条公理拯救。我们假设用户不需要把算符叫做 multiple-value-bind 或 invoke-restart-interactively 才能记住它们的作用。

另一方面，我们也尽量不让这门语言看起来像一个正在咒骂的卡通人物。你有没有注意过，当你填写某种 Web 表单时，往往会全部使用小写？普通的英文字母比 #&%$ 之类的字符打起来省力得多。（Dan Giffin 最近观察到，如果你按按键次数来衡量 Perl 程序，它们看起来就没那么短了。）而且小写字母也更容易阅读。你读符号时必须按一种类似精神上的 Shift 键。我们会尽可能让 Arc 成为一门流畅的小写语言。

很多优秀程序员还有一个共同点，就是初期腕管综合征，所以一门容易输入的语言对他们来说应该是个加分项。

Arc 的多态性很强。+ 函数既可以相加数字，也可以拼接字符串。为每种情况设置单独的算符，等价于只有一个算符外加一个类型声明。而使用单个算符则等价于允许程序员省略声明。

每门语言都应该和一个用它编写的大型应用一起设计，这样设计者才能看到语言是否真的好用。例如，C 就是在最终形成 Unix 的系统编程项目中被磨砺出来的。我们打算用 Arc 来编写一个基于 Web 的应用平台。两者将会紧密集成，就像 C 和 Unix 一样。我们希望让 Arc 成为编写 Web 应用最理想的语言——也就是我们当年写 击败平均水平 时希望拥有的语言。这意味着 Arc 必须善于处理字符串，而这在过去并不是 Lisp 特别关注的问题。

Perl 让程序变短的办法之一是大量使用代词。Lisp 程序员一直通过 aif 之类的宏在一定程度上这样做，但通常被认为是一个有点可疑的技巧，据我所知并没有进入任何主流 Lisp 方言。在 Arc 中我们会大量使用代词。它们让程序更短，而且拥有它们能给程序员更多控制权。（如果一门语言允许事物隐式，那么程序员总是可以选择显式；但如果语言要求一切都必须显式，程序员就无法让事物隐式了。）

5. 语法

- CL/Scheme：只使用 s 表达式。

缺点：冗长

- Dylan/Python：s 表达式隐藏在下面。

缺点：宏不自然

- Arc：语法作为缩写。

缺点：目前还没有语法

Arc 将会拥有语法。Lisp 社区为此争论了很长时间。事实上从一开始就是这样，正如好几个人在 LL1 会议上指出的那样。McCarthy 原本打算让最初的 Lisp 拥有语法，但程序员更喜欢直接使用原始的 s 表达式，所以从来没人去实现它。语法问题

支持语法的理由通常是它能让 Lisp 对“主流”受众更易接近。Dylan 的设计者本打算把它做成给大众用的 Lisp，于是在后来的版本中 dutifully 加上了一层语法。我没有感觉到他们自己更喜欢用新语法写程序。那是一片危险地带。

通过决定让 Arc 成为一门为优秀程序员设计的语言，我们至少得到了这个问题一侧的答案。我们不必把语言变傻来让任何人容易上手。然而，语法还有另一个优势：它能让程序更短。而这确实是一个真正的胜利。

因此，我认为对于这个长期思考的 Lisp 语法问题的答案（或一个答案）是：是的，要有语法，但仅作为缩写。Arc 将会有语法，但它会以清晰定义的（实际上是可重定义的）方式翻译成底层的 s 表达式。几乎所有语法都是可选的，而且是在单个算符级别上可选。

我们会尽可能让用来定义语法的机制对程序员开放，这样你就可以随心所欲地定制语法。Lisp 被广泛认为是领域特定小语言最好的基础，而可编程的语法应该能让它变得更好。

这种做法的缺点是我们目前还没有语法示例可以展示给大家。我们有一些想法，但还在完善底层的 s 表达式语言。有些事情相当可预测，比如中缀数学。但我们想让 [ 和 ] 这样令人满意的字符发挥最大价值，所以可能会先观察最常见的习惯用法，再决定要缩写什么。

这里是几个想法：

x.y 和 x:y 分别代表 (x y) 和 (x 'y)。

[+ _ 1] 代表 (fn (x) (+ x 1))

我们还计划允许程序员在不会产生歧义的地方省略括号，并用缩进而不是括号来显示结构。我发现自己在白板或信封背面手写 Lisp 时，会自发地做这两件事。

6. Arc 核心

- eval、car、cdr、cons、quote

- cond 分拆为 cond + do（相当于 progn）

CL: (cond ((a x) (princ "!") (b x)) ((c x) (d x)) (t (e x)))

Arc: (cond (a x) (do (pr "!") (b x)) (c x) (d x) (e x))

- 通常使用 if，它会绑定 it：(if (a x) (car it))

Arc 的核心与 McCarthy 最初的Lisp 的根源的核心基本相同。当应用于符号和列表（1960 年论文中仅有的数据类型）时，算符 eval、car、cdr、cons 和 quote 的行为相同，只是 car 和 cdr 作用于 nil 时会产生错误。

我们唯一修改的算符是 cond。McCarthy 希望把他的公理保持在最少，把 progn 埋在了 cond 里面。这对他的示例有效，但对于一般编程，你很快就会发现需要一个独立于 cond 隐式 progn 的 progn。

在 cond 中有隐式 progn 意味着每个 cond 子句都必须多加一对括号。McCarthy 后来说他认为自己把 cond 设计错了，它用了太多括号，这可能就是他所指的意思。Arc 的 cond 没有隐式 progn，所以你不需要在每个子句周围加括号。我们还省略了默认子句中的 t，它看起来也是一颗洋葱。幻灯片中的例子展示了 Common Lisp 和 Arc 中相同的代码。（Arc 的 do 相当于 Common Lisp 的 progn，pr 相当于 Common Lisp 的 princ。）

在 Arc 中，cond 是一个低级算符，主要用于宏展开。大多数时候程序员使用 if，它的功能完全一样，只是在成功的 then 表达式中，变量 it 会被绑定到测试表达式的结果上。

7. 赋值（作用域）

- 词法作用域，单一命名空间。

- = 是 setf（没有 set 或 setq）。

- 可以通过赋值创建局部变量。

> (do (= x 5) (cons x 'a)) (5 . A)

- 任何代码序列都是一个块。

- 很容易写出不影响作用域的块。

Arc 像 Scheme 一样拥有词法作用域和单一命名空间。一个值为函数的变量与其他变量没有区别。求值规则很简单：从左到右求值整个表达式，然后把第一个元素的值应用到其余元素的值上。

赋值算符是 =。我本来对此持怀疑态度，但决定试试看自己是否能习惯。结果它工作得很好，即使在前缀形式中也是如此。条纹很显眼，这就是为什么它们被用在警告标志和有毒动物身上。

Arc 与以往 Lisp 的一个重大区别是：局部变量可以通过赋值隐式创建。如果你对一个尚不存在的变量进行赋值，你就 thereby 创建了一个词法变量，它会持续到所在块的结束。（是的，我们知道这会让代码难以编译，但我们打算试试。）一个块是一个 do，或者 Arc 预定义算符中任何一个隐式的 do。

很容易写出一个类似 progn 但不影响作用域的算符：只要写一个接受任意数量参数并返回最后一个的函数即可。Arc 有一个目前叫 justdo 的函数，它就是干这个的，但它仅供宏内部使用——当你必须顺序求值一系列表达式而又不希望它们意外地处于新的词法环境中时。

8. 函数和宏

- lambda 叫 fn：(fn (x) (cons x 'a))

- 用 rfn（一个宏）代替 labels。

(rfn len (x) (if (no x) 0 (+ 1 (len (cdr x)))))

- 宏与一等对象分离。

(macro (test . body) `(if ,test (do ,.body)))

- 要获得局部宏，只要绑定即可。

像任何 Lisp 一样，Arc 拥有一等函数。有一个 fn 算符，类似于 Scheme 的 lambda，用于返回一个新函数。Arc 没有 labels 或 letrec。定义递归函数时有一个叫 rfn 的宏，它类似 fn，只是额外接受一个第一个参数作为它自己的名字。上面的例子是一个求列表长度的递归函数。

使用 rfn 而不是 labels，使得定义单个递归函数更方便，而定义多个相互递归的函数则不太方便，但前者显然是更常见的情况。

（rfn 这个名字是 Dorai Sitaram 建议的。）

Arc 的宏也是第一等对象。它们像 Common Lisp 中一样，仅仅是返回表达式的函数。Arc 没有 Scheme 或 Dylan 的卫生宏。或者更准确地说，它不强制你使用它们；类似的功能可能会以库的形式提供，但程序员也可以接触到原始的宏展开。

这又是一个“用户是优秀程序员”这条公理简化事情的例子。作为专家，用户不会被变量捕获的前景吓倒（事实上常常会故意这么做），但肯定不喜欢一个会夺走 defmacro 部分能力的宏机制。

因为宏是第一等对象，所以不需要 Common Lisp 的 macrolet。你可以用 let 给一个宏局部作用域，就像给任何其他变量赋值一样。

让宏成为第一等对象可能会严重影响编译。我们希望通过类型推断和声明，能够在需要时得到快速的代码。

宏在 Arc 中将会是一个重点，因为我们认为它们是 Lisp 最大的优势之一。除了传统的基于表达式的宏，Arc 可能还会拥有由代码遍历器驱动、能查看多个表达式的宏。这可能会与 Arc 的可编程语法以有用的方式进行交互。

9. 绑定

- with 类似 CL 的 let：

> (with (x 'a y 'b) (list x y)) (A B)

- let 用于单个变量的情况：

> (let x 'a (cons x 5)) (A . 5)

- 两者都是函数调用的宏

Arc 有一个引入新变量绑定的宏，叫 with。它类似于 Common Lisp 和 Scheme 的 let，只是用了更少的括号。

在我用 Scheme 和 Common Lisp 写的代码中，大多数 let 只引入一个变量。所以在 Arc 中，我们把 let 这个名字留给这个更常见的情况。

with 和 let 都是基于函数应用的显而易见的宏。例如，(let x 3 (foo x)) 展开成 ((fn (x) (foo x)) 3)。

只要可能，像这样可以用 Arc 本身实现的算符，都会用一段 Arc 源码正式定义它们。它们不必非得这样实现，或者在教程中这样描述，但如果一门编程语言是好的，那么源代码应该是最好的规格说明。

10. 迭代

- CL 的 do 难以阅读。解决常见情况。

- Arc 的 4 个基本迭代器：

> (for (= i 0) (< i 10) (++ i) (pr i)) 0123456789 NIL

> (to i 10 (pr i)) 0123456789 NIL

> (each x '(a b c) (pr x)) ABC NIL

> (let i 0 (while (< (++ i) 10) (pr i))) 123456789 NIL

- 像 with 和 let 一样，是函数调用的宏

迭代的形式在 Lisp 社区中是另一个悬而未决的问题。传统的 Lisp do 虽然通用，但太难阅读。如果 do 是一种表达迭代的自然方式，我到现在应该已经习惯了，但每当我看到一个 do，我都得停下来解码它；而当我写一个 do 时，我也得停下来思考如何表达我想要的迭代。换句话说，do 感觉像是目标代码。

我们对迭代的看法是：存在少数几种常见情况。如果你直接支持这些情况，你就能覆盖几乎所有人实际写的迭代。如果有人非常想念 do，他们随时可以把它写成一个宏。

Arc 有四个迭代算符：for，类似于 C 的 for，只是初始化形式创建的绑定是循环体内的局部变量；to，类似于 Common Lisp 的 dotimes，但没有返回值的形式；each，类似于 Common Lisp 的 dolist，但可用于任何复合数据对象（列表、字符串、向量等），同样没有返回值的形式；以及 while，当测试表达式返回真时就执行它的循环体。

所有这些迭代算符都是（虽然不一定这样实现）基于递归函数应用的显而易见的宏。

11. 迭代捕获

- while 会捕获 it：

(while (read) (pr it))

- 所有迭代算符都会捕获 keep 和 sum：

> (each x '("al" "bob" "joe") (if (> (len x) 2) (keep x))) ("bob" "joe")

> (to x 5 (sum x) (pr x)) 01234 10

（不能同时使用两者。）

像 if 一样，while 会让变量 it 绑定到测试表达式返回的值上。

此外，所有迭代算符都会在循环体内让 keep 和 sum 绑定到用于累积值的函数。调用 keep 会累积一个值的列表，调用 sum 会累积一个总和。你可以在循环中的任何位置、以任意频率调用它们，但不能在同一个循环中同时调用 keep 和 sum。如果你调用了 keep 或 sum，迭代表达式将返回累积的值，否则返回 nil。

我们可能会把累积泛化，允许对累积值应用任意函数，并可能把 keep 和 sum 实现为基于这个机制的宏。

12. 数据类型

Symbol Number（与 CL 相同）Cons Character String Array Class, Obj DB（hash/alist）Function Macro 可能还会更多

这是目前数据类型的快速列表。大致符合你的预期。一旦我们开始处理线程、异常之类的东西，很可能会添加更多类型。

13. 复合对象 = 作用于索引的函数

> ("hello" 2) \l

> (map "carpet" '(3 4 1 2)) (\p \e \a \r)

CL: (aref a 5) C: a[5] Arc: (a 5)

任何复合数据对象（即拥有多个可分别寻址部分的）都表现得像一个作用于索引的函数。因此，例如要获取列表的第三个元素，你就“调用”这个列表并传入 2 作为参数。这让程序更短，也省去了为每种数据类型单独写访问函数的麻烦。

你可以真正地在任何能使用函数的地方使用复合数据对象，包括作为 map 的第一个参数（类似于 Common Lisp 的 mapcar，但可用于任何序列）。

14. 字符串的行为像列表

> (car "abc") \a > (cons \a "bc") "abc"

像整数和浮点数那样的传染

> (+ "abc" '(d e)) (\a \b \c D E)

我们是否应该允许字符串中包含非字符？

这是一个有点激进的想法，但我们决定试试看效果如何。在 Arc 中，字符串的行为像列表：字符串的 car 是一个字符，cdr 是去掉第一个字符后的另一个字符串。

我们发现对列表的递归是一种非常有用的技术，所以也许它对处理字符串也同样有效。

这当然可能效率极低，我们计划让程序员可以在需要时声明某些字符串不需要这样处理，应该表示为简单的连续内存块（即 unboxed）。

这引出了一个有趣的可能性。当字符串实际上是列表时，为什么不让程序员往里面插入任意对象呢？例如，插入一个拥有自己 pr 方法的对象可能会很有用。目前字符串只能包含字符，但我们走着瞧。

当你把字符串和列表结合时，会得到类似浮点数传染的东西。字符串是一种特殊的列表，就像整数是一种特殊的实数，当你把它们结合时，得到的结果是更一般类型的。

15. 类和对象

单继承（可能会改变）

(= pt (class nil 'x 0 'y 0))

(type pt (x 0) (y 0))

(= p1 (new pt))

> (p1 'x) 0 > (++ (p1 'x)) 1

我们把面向对象编程看作一种常常有用的抽象，而不是目的本身。一个特性的真正检验标准是它是否真的能让你的源代码更短、更简单。至少在开始时，Arc 有一个最小的对象系统，能让你做我们知道人们需要做的事情。

类是第一等对象，你可以用 class 算符创建它。它接受一个父对象（或 nil），后面跟着一系列字段名与默认值交替出现的列表。字段名可以是任何东西，不仅仅是符号。还有一个 type 宏，会展开成对 class 的调用，用于最常见的无父类且字段名为符号的情况；其语法非常像 Common Lisp 的 defstruct。

目前只有单继承。要支持多继承并不难，但我们询问了一些杰出的黑客朋友，他们中没有一个人觉得它特别有用。

你可以通过对类调用 new 来创建一个实例。你引用实例（或类，如果你愿意的话）中的字段的方式，与引用任何复合数据结构的一部分的方式相同：把对象当作函数来调用，传入索引，这里就是字段名。

16. 重载

- 任何东西都可以作为对象的字段名。

- 通过使用一个 fn 作为名字来进行重载。

(= pt (class nil 'x 0 'y 0 pr my-pr))

- 意味着对第一个参数进行派发。

- （对 cons 不太好，所以也许需要一种方式来声明函数的关键参数。）

你可以通过给一个对象一个字段——其名字就是某个函数（实际的函数对象）——来重载该函数。当一个表达式被求值时，如果第一个参数是一个实例（或类），并且它有一个字段的名字正是那个函数，那么存储在该字段中的值（它也必须是一个函数）会被调用，传入相同的参数。

在幻灯片的例子中，pt 是一个打印方式不同的类。当 pr 被调用在一个 pt 的子类上时，就会改而调用 my-pr（不管它是什么）。

这个重载方案本质上是对第一个参数进行派发。这对 cons 这样的情况不太好，因为它的关键参数实际上是第二个。也许 cons 是个特例（它的参数顺序是出于视觉原因，而且这种情况应该很少见）。如果不是，我们可能会增加一种方式来说明一个函数的关键参数是哪一个。

17. 数据库是哈希表或关联列表

(newdb eq 'x 'a 'y 'b)

(db x 'a y 'b)

> (each x (db x 1 y 2) (pr x) (keep key)) 12 (X Y)

查找失败返回 *fail*

Arc 有一个叫做 db 的数据仓库，你可以把它想象成一个哈希表，虽然其内部表示未作规定（在某些情况下编译器可能会选择让它成为一个关联列表）。

（这个想法是 Erann Gat 提出的。）

db 的键可以是任何类型的对象，就像实例的字段名一样。不过，你可以随时添加或删除条目，也可以指定用于定义键相等性的函数。

你可以通过调用 newdb 来创建一个 db，或者用更短的 db，它假设查找相等性测试是 eq 且键是符号。

当你进行查找时，如果什么都没找到该怎么办？传统的 Lisp 答案（如 assoc）是返回 nil，但在这种情况下，你如何区分“没找到”和“找到了 nil”？Arc 的答案是使用一个全局变量 *fail*，由没有找到匹配的查找函数返回。它默认绑定到 nil，这在几乎所有时候都是正确的；当不是的时候，你可以用一个 let 把 *fail* 绑定到一个 gensym。

18. 参数列表

参数可以是符号或 (opt | get | ds ...)

(def foo (x (ds (i j)) (get m n) (opt q 'a) . z) (list x i j m n q z))

> (foo 1 '(red green) (db m 'a n 'b) 'hel 'lo) (1 RED GREEN A B HEL (LO))

（语法在这里会有帮助。）

> (let (ds (x y)) '(a b) (list x y)) (A B)

Arc 允许在参数列表中除了普通符号外还有三种东西：(opt var default) 表示一个可选参数，其默认值为 default；(ds pattern) 表示用一个变量模式去匹配传入的列表，类似 Common Lisp 的 destructuring-bind；以及 (get vars) 表示从传入的 db 或实例中按对应键提取一个或多个变量。

我们期望 get 参数能扮演 Common Lisp 中关键字参数的角色，并希望把调用编译成类似的代码（即实际上不创建 db）。

Arc 也支持 rest 参数，它出现在参数列表中一个点号之后，会被赋予调用中剩余的所有值。

因为我们去掉了 Common Lisp 参数列表中那些额外的非变量记号，所以我们可以把 let 和 with 定义为基于函数调用的显而易见的宏。因此我们免费获得了相当于 Common Lisp destructuring-bind 的功能：只要在 let 或 with 的参数中使用 ds 形式即可。用 get 形式对实例进行解构也会很方便。

19. 速度

- Moon：很难分辨什么东西是昂贵的。

- SICP：程序“供人阅读”

- 应该由 profiler 告诉我们，而不是语言。

- 尤其是对基于服务器的应用来说。

David Moon 曾经告诉我，Lisp 让程序员很难分辨什么操作是昂贵的。这听起来是个问题。与此同时，Abelson 和 Sussman 说（我也同意）“程序必须写给人读，只是附带地让机器执行。”

我们如何调和这两个想法呢？我同意，最重要的是，一门语言必须是思考的好工具。这正是 Lisp 当初好的原因。然而，像任何人一样，我也喜欢快的代码。我认为出路是把显示什么东西昂贵的负担从语言身上拿走。不要试图让语言暗示什么东西是昂贵的，而是让语言便于表达思想，然后用一个 profiler 来显示什么东西是昂贵的。

Scheme 语言偷偷扩大了语言设计者的权力范围。从很早的时候，也许从一开始，Scheme 规范就说符合要求的实现必须进行尾调用消除。我第一次读到这个的时候，我想“等等，你能在规范里要求这个吗？”Arc 将会看到这种扩大，并通过一些性能分析的标准来进一步提高。

在 Arc 中获得快速代码的方式是先对它进行性能分析，然后添加声明，在需要的地方提高效率。像 Common Lisp 一样，声明是对编译器的优化建议；它们不应该影响程序的含义。

Arc 旨在用于另一条前路，而性能分析在那里尤其有用，因为你可以分析程序的真实行为。你不需要依赖测试套件；你可以观察真实的用户。
